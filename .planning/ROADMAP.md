### Phase 71: Spring Boot 4.0.6 Upgrade + Thymeleaf 3.1.5 Template Audit + Build Guard

**Goal**: Spring Boot 4.0.5 → 4.0.6 platform debt is eliminated before any backup feature code is written. The transitive Thymeleaf 3.1.5 (CVE-2026-40478 SpEL canonicalization hardening) is absorbed via controller-side `pageTitle` model attributes on the 3 known broken templates plus every additional offender uncovered by a mechanical audit of all ~80 admin + site templates. A `TemplateRenderingSmokeIT` covers all `/admin/**` GET routes against `org.thymeleaf.exceptions.TemplateProcessingException`, and a lightweight `exec-maven-plugin` grep gate prevents regression of the fragment-parameter-ternary pattern.

**Depends on**: Nothing (first phase of v1.10; clean baseline from v1.9 close)

**Requirements**: PLAT-01, PLAT-02, PLAT-03, PLAT-04, PLAT-05, PLAT-06, PLAT-07

**Success Criteria** (what must be TRUE):

  1. `pom.xml` declares `spring-boot-starter-parent` 4.0.6 and the `<dependencyManagement>` block pins `org.thymeleaf:thymeleaf` to 3.1.5
  2. All 3 known templates (`match-scoring-form.html`, `race-scoring-form.html`, `season-phase-form.html`) plus every additional fragment-parameter expression discovered by the audit pass `${pageTitle}` from the controller — no ternaries or expressions remain in `th:replace`/`th:insert`/`th:include` parameter positions
  3. `./mvnw verify -Pe2e` is BUILD SUCCESS on 4.0.6 with all 1227 unit + 31 Playwright E2E tests green; JaCoCo line coverage stays ≥ 82 %
  4. `TemplateRenderingSmokeIT` GETs every `/admin/**` route with dev-data-seeded fixture and asserts HTTP 200 + absence of `TemplateProcessingException` in the response body
  5. The Maven build fails when a contributor reintroduces the pattern `th:(replace|insert|include)=".*\(.*\$\{.*\?.*:.*\}.*\)"` anywhere under `src/main/resources/templates/`

### Phase 72: Backup Wire Contract — Schema, Manifest, ObjectMapper, Audit-Log Scope

**Goal**: Lock the on-disk wire contract for the backup ZIP before any export or import code is written. `BackupSchema.SCHEMA_VERSION = 1` (integer, GAP-2 resolution), `BackupManifest` record (schema_version / app_version / export_date / table_counts), `BackupObjectMapperConfig` `@Qualifier("backupObjectMapper")` bean (isolated from default Spring `ObjectMapper`), and Flyway V7 `data_import_audit` table all land in this phase. The 22-entity FK-respecting `EXPORT_ORDER` list is generated from the live entity classes (GAP-5: not hand-written). The decision to keep `data_import_audit` strictly out of the export scope is documented in PROJECT.md Decisions. The per-entity ZIP layout (`data/<entity>.json` + `manifest.json` first, `uploads/` mirror — GAP-1 resolution) is documented as the canonical wire contract.

**Depends on**: Phase 71 (clean 4.0.6 baseline; Thymeleaf 3.1.5 audit complete so admin pages can render the future preview screen safely)

**Requirements**: SCHEMA-01, SCHEMA-02, SCHEMA-03, SCHEMA-04, IMPORT-08

**Success Criteria** (what must be TRUE):

  1. `BackupSchema.SCHEMA_VERSION = 1` is exported as a public static `final int` constant; `BackupSchema.EXPORT_ORDER` is a generated FK-respecting `List<EntityRef>` covering all 22 operative entity classes
  2. `BackupManifest` record serializes to a `manifest.json` that is provably the FIRST entry in any ZIP produced by the future export service (read-order determinism enforced in the writer)
  3. Flyway V7 `data_import_audit` migration runs cleanly on H2 + MariaDB profiles; `data_import_audit` table is created with the documented columns (id UUID PK, executed_at, executed_by, schema_version, table_counts_wiped JSON, table_counts_restored JSON, source_filename, success)
  4. `@Qualifier("backupObjectMapper")` bean is injectable and behaves as configured (`FAIL_ON_UNKNOWN_PROPERTIES=true`, ISO-8601 dates, JavaTimeModule, MixIns registered) without polluting the default Spring `ObjectMapper` (verified by an IT that injects both qualifiers and asserts they are distinct beans)
  5. PROJECT.md Decisions section contains a row stating "audit log table `data_import_audit` is permanently out of export scope" with rationale; per-entity ZIP layout (`data/<entity>.json`) and integer schema versioning are documented as canonical wire-contract choices

### Phase 73: Backup Export — Jackson MixIns + Streaming ZIP Endpoint

**Goal**: A single admin click on `/admin/backup` produces a streamed ZIP download whose contents round-trip cleanly through the manifest contract from Phase 72. The ~22 per-entity Jackson MixIns under `org.ctc.backup.serialization` apply `@JsonIdentityInfo` externally — no annotation drift in `org.ctc.domain.model`. `BackupExportService` is `@Transactional(readOnly=true)` with explicit `@EntityGraph` eager-fetch on every collection field reachable in the export aggregate, eliminating `LazyInitializationException` during streaming. Export is delivered via `StreamingResponseBody` (no full-dataset memory buffering) with a CSRF-protected POST endpoint and a `Content-Disposition` filename that includes an ISO-instant timestamp.

**Depends on**: Phase 72 (needs `backupObjectMapper` bean, `BackupSchema.EXPORT_ORDER`, and `BackupManifest` to write any ZIP at all)

**Requirements**: EXPORT-01, EXPORT-02, EXPORT-03, EXPORT-04, EXPORT-05, EXPORT-06

**Success Criteria** (what must be TRUE):

  1. Admin clicks the `Backup` sidebar entry and lands on `/admin/backup`; clicking `Export Backup` triggers a `POST /admin/backup/export` that streams a ZIP file with `Content-Disposition: attachment; filename=ctc-backup-<ISO-instant>.zip`
  2. The downloaded ZIP contains `manifest.json` as the FIRST entry, per-entity JSON files under `data/` (e.g. `data/seasons.json`, `data/race-results.json`), and an `uploads/` directory mirroring only entity-referenced files from `data/{profile}/uploads/`
  3. Domain entity classes under `org.ctc.domain.model` are byte-identically unchanged after this phase — all serialization concerns live in `org.ctc.backup.serialization` MixIn classes
  4. `BackupExportService` runs under `@Transactional(readOnly=true)` and produces a streamed export of the dev fixture (Saison 2023 + 2024-3) without a single `LazyInitializationException` in the logs
  5. Anonymous (unauthenticated) `POST /admin/backup/export` is rejected by Spring Security on `prod`/`docker` profiles; CSRF token is required and verified

### Phase 74: Backup Import Preview + ZIP Hardening + Multipart Config + Schema-Version Gate

**Goal**: A no-write import path that reads the manifest, validates the schema version, and renders a per-table wipe + restore preview — without ever touching the database. ZIP-Slip and ZipBomb defenses harden the multipart upload (per-entry 50 MB / total 500 MB / 50.000 entries cap; `startsWith(uploadDir.toRealPath())` traversal check reusing the v1.1 SECU-02 defense). Multipart limits are raised to 100 MB on Spring AND Tomcat layers (Tomcat's `max-http-form-post-size` and `max-swallow-size` overlap-aware). `MaxUploadSizeExceededException` is mapped in `GlobalExceptionHandler` with a friendly Flash message. The schema-version mismatch is rejected as HTTP 400 + Flash before any DB-write transaction begins (catastrophic-data-loss prevention).

**Depends on**: Phase 73 (round-trip needs an export to validate the import preview against; preview re-uses `BackupManifest` + `backupObjectMapper`)

**Requirements**: IMPORT-01, IMPORT-02, IMPORT-03, IMPORT-04, SECU-01, SECU-02, SECU-03, SECU-04

**Success Criteria** (what must be TRUE):

  1. Admin uploads a ZIP via `POST /admin/backup/import-preview`, lands on a preview screen with per-table current-rows vs imported-rows + uploads-file count + schema-version-match indicator
  2. A ZIP whose `manifest.schema_version` ≠ `BackupSchema.SCHEMA_VERSION` is rejected with HTTP 400 + an admin-readable Flash message; an IT proves the database is byte-identically unchanged after the rejection
  3. A ZIP containing a path-traversal entry (`../../etc/passwd`), an absolute path, or a per-entry size > 50 MB is rejected with a Flash message; a malicious test fixture is committed to verify
  4. An upload exceeding 100 MB triggers `MaxUploadSizeExceededException` and renders the user-facing Flash message "Backup-Datei überschreitet die Maximalgröße von 100 MB. Bitte komprimieren oder kontaktieren Sie den Admin." instead of a Tomcat stack trace
  5. Preview state is STATELESS — re-parse on execute via the staging-path pattern; no `@SessionAttributes`, mirroring v1.8 D-15

### Phase 75: Replace-All Transaction + JPA Auditing Bypass + Live MariaDB UAT

**Goal**: The riskiest phase. A single `@Transactional` boundary wipes all 22 operative tables in FK-reverse order via `EntityManager.createNativeQuery("DELETE FROM ...")` (TRUNCATE auto-commits on MariaDB → forbidden), drops the L1 cache via `em.flush() + em.clear()`, then bulk-restores via `JdbcTemplate.batchUpdate` (bypassing `AuditingEntityListener` so imported `created_at`/`updated_at` survive verbatim). The `Team.parentTeam` self-FK is decoupled in a pre-step (`UPDATE teams SET parent_team_id = NULL`). Upload-tree restore is post-commit (file-system mutations are NOT in the JPA transaction) via stage-and-rename, leaving the previous tree at `data/.import-backups/<ts>/uploads-old/` for 24 h manual recovery. A `BackupImportRollbackIT` injects an exception at 50 % of the restore loop and asserts the DB is in pre-import state with `success=false` audit row. Live UAT against the Saison-2023 fixture on local MariaDB confirms standings + driver-ranking + phase-breakdowns are byte-identical after a full export → wipe → import cycle.

**Depends on**: Phase 74 (needs the schema-version-validated, hardened ZIP reader from preview before write semantics are enabled)

**Requirements**: IMPORT-05, IMPORT-06, IMPORT-07, QUAL-03

**Success Criteria** (what must be TRUE):

  1. Admin confirms the preview, the import runs to completion, and the database state matches the export's row counts byte-identically per `data_import_audit.table_counts_restored`
  2. Imported entities preserve their original `created_at` / `updated_at` / `version` values verbatim (an IT exporting a row with `created_at=2024-01-01`, wiping, and importing asserts the value is still `2024-01-01` after re-read)
  3. A mid-restore-failure injection rolls back to the pre-import database state with no orphan rows in any of the 22 tables; `data_import_audit` records `success=false` with the failure stack trace in the SLF4J log
  4. The previous `uploads/` directory tree is preserved at `data/.import-backups/<ts>/uploads-old/` after a successful import; the new tree is in place atomically (no half-written files visible to readers between the rename steps)
  5. Live MariaDB UAT (Saison-2023 driver fixture) confirms via the public site that standings, driver ranking, and phase breakdowns are visually identical after export → wipe → import

### Phase 76: Operational Hardening — Import Lock + Read-Only Banner + Auto-Backup-Before-Import

**Goal**: Defense-in-depth around the Replace-All path. `ImportLockService` (`@Service @Scope("singleton")`) holds a `ReentrantLock`; a second concurrent import attempt is rejected with HTTP 409 + Flash. While the lock is held, a yellow read-only banner renders on every admin page ("Import läuft — Schreibzugriff temporär gesperrt"); a `@ControllerAdvice` filter rejects any non-import POST under `/admin/**` with HTTP 503. Before the wipe transaction begins, `BackupImportService` synchronously triggers an auto-export to `data/.import-backups/<ts>/auto-backup-before-import.zip` — if the auto-export fails, the import aborts before any DB mutation. This is the safety net for "live UAT with real data" runs against MariaDB.

**Depends on**: Phase 75 (lock + banner + auto-backup wrap the existing wipe + restore path)

**Requirements**: SECU-05, SECU-06, SECU-07

**Success Criteria** (what must be TRUE):

  1. While import A is running, attempt B → HTTP 409 + Flash "Ein anderer Import läuft bereits — bitte warten"; verified by a 2-thread integration test with a deliberately slow import
  2. While the import lock is held, every admin page renders the persistent yellow `Import läuft` banner; a non-import POST under `/admin/**` returns HTTP 503; the import-execute route itself is whitelisted and proceeds normally
  3. The auto-backup ZIP is present at `data/.import-backups/<ts>/auto-backup-before-import.zip` after every (successful or failed) import attempt; if the auto-export step itself fails, the import is aborted with no DB mutation and a clear Flash message
  4. The 24 h retention of `data/.import-backups/<ts>/uploads-old/` and the auto-backup ZIP is documented in the operational runbook (text file, not a UI feature)
