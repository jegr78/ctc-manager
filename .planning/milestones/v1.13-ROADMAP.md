## Milestone Goal

Eliminate manual Discord work for the CTC league operator by integrating channel-creation, role-permission setup, structured post-publishing, and post-edit workflows directly into the CTC Manager admin app. The app becomes the single command center for all Discord-side league communications. Bundles three v1.12 audit carry-forwards plus a bookkeeping-glitch fix as the first phase so v1.13 starts on a clean baseline before Discord migrations land.

**Estimated duration**: 2-3 days
**Success Criteria** (what must be TRUE):

### Phase 94: Team Roles + Match Channel Lifecycle
**Goal**: Operator can map teams to Discord roles, create per-match Discord channels with full permission-overwrite model, and archive them via a category-picker modal honoring Discord's 50-channels-per-category limit.
**Depends on**: Phase 93 (Discord client + sealed exception hierarchy + config page must exist before channel-creation buttons can be wired; CHAN-02 requires INFRA-01's `DiscordRestClient.createChannel` + `createWebhook` methods)
**Requirements**: CHAN-01, CHAN-02, CHAN-03
**Estimated duration**: 2-3 days
**Success Criteria** (what must be TRUE):
  1. Team-Form `discordRoleId` field round-trips through repository IT, rejects non-snowflake input via the `^\d{17,20}$` validator, and offers a live-dropdown of guild-roles when the Discord-Bot is reachable (sourced from `DiscordRestClient.fetchGuildRoles()` 60-min cache); operator can clear the field to disable Discord-channel-creation for that team.
  2. Creating a match channel produces correct permission-overwrites verified by a post-create permission-audit assertion (Bot fetches the just-created channel, asserts only the 2 whitelisted team-roles have `VIEW_CHANNEL`, any other role with View triggers `DiscordAuthException`); channel name follows `md{N}-{teamA.shortName}-vs-{teamB.shortName}` (lowercase, dash-separated, Discord-enforced).
  3. "Create Discord Channel" button visibility is gated on `match.discordChannelId == null` AND both teams have `discordRoleId` AND season has current-category set; on click the channel is created with the permission bitmasks from `DiscordPermissions.TEAM_MEMBER_ALLOW_MASK` / `TEAM_MEMBER_DENY_MASK`, a webhook is created in the new channel, and both IDs are stored on the match row.
  4. Archive modal lists only categories where `year == match.season.year` (regex match `^Match Days Archive (?<year>\d{4})(?: \((?<num>\d+)\))?$`, sorted ascending by `num`), displays live channel-count as `current/50`, defaults to highest-num category with `< 50` channels, and confirms-and-moves the channel via Bot PATCH `parent_id`; when all categories are full, `DiscordCategoryFullException` renders with the `category-full` badge and links to the operator-runbook section "Creating a new archive category".
  5. Flyway V9 (`teams.discord_role_id VARCHAR(32)`) + V10 (`matches.discord_channel_id`, `discord_channel_webhook_url`, `discord_teaser`, `stream_link`, `lobby_host`, `race_director`, `streamer`) migrations apply cleanly on H2 + MariaDB.

### Phase 95: Match Channel Posts
**Goal**: Five per-match post types (Team Cards, Settings, Lineups, Schedule, Match Results) post via Webhook with stored `message_id` and a uniform in-place edit path via Webhook-PATCH.
**Depends on**: Phase 94 (a match channel + webhook URL must exist on the match row before posts can target it; `DiscordPostService.postOrEdit` lookup keys on `match.discordChannelId`)
**Requirements**: POST-01, POST-02, POST-03, POST-04, POST-05
**Estimated duration**: 2-3 days
**Success Criteria** (what must be TRUE):
  1. Re-posting a graphic edits the existing Discord message in place — Discord's edit-indicator is visible, no duplicate appears, and the `discord_post` row's `updated_at` advances (verified by IT for all 5 post types: `TEAM_CARDS`, `SETTINGS`, `LINEUPS`, `SCHEDULE`, `MATCH_RESULTS`).
  2. "Post Team Cards" issues ONE multipart Webhook-POST containing both team-card PNGs as `files[0]` + `files[1]`; WireMock body-pattern assertion confirms a single multipart body carries both attachments and a single `message_id` is stored in the resulting `TEAM_CARDS` row.
  3. "Post Match Results" button label changes to "Update Match Results" when an existing `MATCH_RESULTS` post is detected via `discord_post.updated_at` vs `match.lastModifiedAt`; stale-detection only triggers when underlying data has actually changed (no PATCH on a no-op save).
  4. Schedule embed auto-edits on Match-Form save when any of `lobbyHost` / `raceDirector` / `streamer` changes AND a `SCHEDULE` post exists; empty host/RD/streamer fields render as `_TBD_`; no edit fires when fields are unchanged (verified by IT covering all three diff branches).
  5. `/admin/discord/posts` page lists all `discord_post` entries filterable by season, match, and post-type; each row has working Re-Edit + Re-Post buttons (verified by Playwright E2E covering: filter by season, filter by post-type, re-edit click, re-post click).
  6. Flyway V11 (`discord_post` with channel_id, message_id, webhook_id, webhook_token, post_type, match_id, matchday_id, race_id, season_id, posted_at, updated_at + 5 FK-indexes per project FK-index convention) applies cleanly on H2 + MariaDB.

### Phase 96: Provisional Graphic + Forum Threads
**Goal**: New `ProvisionalScoresGraphicService` replaces today's manual sheet-screenshot, and operator can link each season's race-results + standings posts to dedicated Discord forum-threads.
**Depends on**: Phase 95 (`DiscordPost` tracking entity + `DiscordPostService.postOrEdit` from POST-01 is reused by FORUM-02 for race-result forum-thread posts; same pattern with `?thread_id=` query param)
**Requirements**: GRAFX-01, FORUM-01, FORUM-02
**Estimated duration**: 2-3 days
**Success Criteria** (what must be TRUE):
  1. Provisional-scores graphic visually matches today's manually-screenshotted Google-Sheets layout (pixel-accurate per the existing screenshot reference, Desktop + Mobile verified via `playwright-cli` per `feedback_playwright_cli`); Thymeleaf template lives at `src/main/resources/templates/admin/provisional-scores-render.html` (following the `*-render.html` convention from other graphic services).
  2. Season-Detail "Discord Integration" section provides 2 thread-linker widgets (race-results + standings); the Link-existing-Thread modal lists active + archived threads (sorted: pinned first, then active, then archived by `last_message_timestamp` desc) via `DiscordForumService.listThreads(forumChannelId)`; the Create-new-Thread modal accepts default-name template (`Season {N} - {year}` for race-results, `{year}` for standings, editable); Unlink clears only the DB field, leaving the Discord thread untouched.
  3. Race-Result posts land in the season's race-results forum-thread (`?thread_id=` query param present on Webhook URL); when the thread is archived, the bot first issues PATCH `/channels/{id}` `archived=false` to unarchive (Discord requires this for posting), then posts, then optionally re-archives per config (default: leave unarchived); applies to both Race-Result and Provisional-Scores forum-thread posts.
  4. "Post Provisional Scores" button on Match-Detail is visible when race-result data exists but match is not yet marked final; clicks render → PNG → multipart POST → `PROVISIONAL_SCORES` row in `discord_post`.
  5. Flyway V12 (`seasons.discord_race_results_thread_id`, `seasons.discord_standings_thread_id`) applies cleanly on H2 + MariaDB.

### Phase 97: Matchday-Level Posts
**Goal**: Three remaining post types (Matchday Pairings + Match Previews batch, Matchday Overview + Power Rankings, Standings) complete the 11-post-type matchday workflow.
**Depends on**: Phase 96 (forum-threads must be linkable on the season before matchday-overview + power-rankings + standings can target them; POST-07 + POST-08 require `season.discordRaceResultsThreadId` + `season.discordStandingsThreadId` to be populated)
**Requirements**: POST-06, POST-07, POST-08
**Estimated duration**: 2-3 days
**Success Criteria** (what must be TRUE):
  1. "Post Match Previews (batch)" iterates over `matchday.matches` and posts one structured Markdown message per match to the announcement-webhook with H1 = `# {season.name}`, H2 = `## Match Day {N}`, H3 = `### {teamA.name} vs. {teamB.name}`, body = `{match.discordTeaser}`, bullet `- Date: <t:N:F>` from first race time via `DiscordTimestamps`, bullet `- Stream: {match.streamLink ?: "TBD"}`, line `Game On! {emoji(teamA.shortName)} {emoji(vsEmojiName)} {emoji(teamB.shortName)}`, attachments `settings.png` + `lineups.png`; emoji resolution uses the `DiscordEmojiCache` from Phase 93 (long-form `<:NAME:id>` syntax required by webhook posts); each iteration creates a `MATCH_PREVIEW` row in `discord_post`.
  2. Auto-edit on Match-Form save fires for `MATCH_PREVIEW` posts when `streamLink` or `discordTeaser` changes (verified by IT covering: streamlink-change → Webhook-PATCH; teaser-change → Webhook-PATCH; both-unchanged → no PATCH).
  3. "Post Matchday Overview + Power Rankings" issues 2 sequential Webhook-POSTs to the season's race-results Forum-Thread (`season.discordRaceResultsThreadId` with `?thread_id=` query param): first `MatchdayResultsGraphicService` (aggregate matchday-overview PNG), then `PowerRankingsGraphicService` (updated power-rankings PNG); both recorded as separate `discord_post` rows (`MATCHDAY_OVERVIEW` + `POWER_RANKINGS`); the button is visible only when all matches in the matchday are final.
  4. "Post Standings to Forum-Thread" posts `StandingsGraphicService` PNG to `season.discordStandingsThreadId` (NOT the race-results thread) — IT asserts the request URL carries the standings thread-id; re-post edits in place when standings change (recorded as `STANDINGS` row scoped to season).
  5. All 11 post types from the brainstorming workflow (`TEAM_CARDS`, `SETTINGS`, `LINEUPS`, `SCHEDULE`, `PROVISIONAL_SCORES`, `MATCH_RESULTS`, `RACE_RESULTS`, `MATCHDAY_PAIRINGS`, `MATCH_PREVIEW`, `MATCHDAY_OVERVIEW`, `POWER_RANKINGS`, `STANDINGS`) have a working button + edit-path; `/admin/discord/posts` filter dimensions cover all of them.

## Out of Scope (Explicitly Deferred)

Full list in `.planning/REQUIREMENTS.md` § "Out of Scope". Highlights:

- Inbound Discord interaction (slash commands, polls, reaction reads) — local app, no always-online endpoint
- Multi-server / multi-guild support — app assumes exactly 1 Discord guild
- Settings-form migration from Google Form — home team continues using existing form
- Auto-trigger on DB events — all posting is button-triggered (operator control)
- Public-site Discord-notification mirroring — out-of-scope
- Per-user timezone override — `app.timezone` is server-global (single-operator model)
