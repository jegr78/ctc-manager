  @PostMapping("/admin/backup/import-execute")
  public String importExecute(
      @Valid @ModelAttribute("confirmForm") BackupImportConfirmForm form,
      BindingResult bindingResult,
      RedirectAttributes ra) {
    if (bindingResult.hasErrors()) { /* unchanged */ return "redirect:/admin/backup"; }
    try {
      backupImportService.reparse(form.getStagingId());                       // unchanged defense-in-depth
      BackupImportResult result = backupImportService.execute(form.getStagingId());
      ra.addFlashAttribute("successMessage",
          String.format("Import completed. %d rows restored across %d tables.",
                        result.restoredTotal(), result.entityCount()));        // D-15 #1
    } catch (BackupArchiveException ex) {
      ra.addFlashAttribute("errorMessage", mapReason(ex));                    // unchanged Phase 74 path
    } catch (UploadsRestoreException ex) {
      // D-15 #3 — DB committed, files failed and were reverted. auditUuid comes from
      // the thread-bound BackupImportException? No — UploadsRestoreException is thrown by
      // BackupImportPostCommitListener AFTER tx commit. The controller cannot easily access
      // the auditUuid here because the listener runs AFTER the controller returns.
      // Resolution: the listener handles its own flash via session or its own redirect?
      // No — the listener runs after the response is committed. The flash strategy is:
      //  - The controller returns BEFORE the listener fires (AFTER_COMMIT happens
      //    AFTER the controller's redirect response is built).
      //  - SO: D-15#3 cannot be a controller flash. Solution: the listener logs ERROR;
      //    the operator interprets the error log; the success flash is still rendered
      //    (because db committed and the listener's UploadsRestoreException does NOT
      //    propagate back to the controller — the controller already returned).
      // CORRECTION: The controller can detect this case ONLY if the listener writes
      // a marker. Since D-15#3 names the soft-fail flash, the design intent is the
      // operator sees the success flash AND inspects the audit row.success=false to
      // detect the uploads soft-fail. The success flash is rendered unconditionally
      // when execute() returns without throwing — that is correct because db is committed.
      // The catch on UploadsRestoreException here is defensive — it never executes
      // in practice (the listener is asynchronous-to-commit-but-synchronous-to-listener-thread,
      // but the listener exception happens AFTER the controller already redirected).
      log.error("UploadsRestoreException in controller path — UNEXPECTED post-Wave 2", ex);
      ra.addFlashAttribute("errorMessage",
          "Import database succeeded but uploads restore failed and was reverted. See logs. Audit-id: unknown.");
    } catch (BackupImportException ex) {
      ra.addFlashAttribute("errorMessage",
          String.format("Import failed and was rolled back — see logs. Audit-id: %s.",
                        ex.getAuditUuid()));                                    // D-15 #2
    }
    return "redirect:/admin/backup";
  }

Planner note on D-15 #3 (CRITICAL):
  RESEARCH §2 Pattern 2 lines 397-405 show the listener throwing UploadsRestoreException
  back to the caller. BUT Spring's AFTER_COMMIT listener invocation: by default, exceptions
  in AFTER_COMMIT listeners do NOT propagate to the calling thread because the listener
  is invoked SYNCHRONOUSLY but AFTER the @Transactional method's return path. The
  controller's response is already on its way back. Therefore:

    - The D-15 #3 soft-fail flash CANNOT be reliably rendered from the controller.
    - Resolution (planner pick, locked here): Render the SUCCESS flash unconditionally when
      execute() returns BackupImportResult. The operator detects the uploads soft-fail
      via (a) the ERROR log line from the listener, (b) data_import_audit.success=false
      written by recordResultBestEffort in the listener's failure paths, and (c) the
      uploaded files in data/.import-backups/<ts>/uploads-old/ NOT having been moved
      back to data/{profile}/uploads/.
    - For consistency, the catch (UploadsRestoreException) clause is added to the
      controller as defense-in-depth (in case a future refactor invokes the move-triple
      from a non-AFTER_COMMIT path), with auditUuid="unknown" placeholder.
    - PATTERNS open question Q5 (UploadsRestoreException GlobalExceptionHandler mapping)
      is hereby RESOLVED: the controller's local catch above is sufficient; no
      GlobalExceptionHandler mapping is added.
