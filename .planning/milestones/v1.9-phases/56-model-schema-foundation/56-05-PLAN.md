Project precedent for persistence integration tests:
- `@SpringBootTest @ActiveProfiles("dev")` (uses H2 in-memory, full Spring context, AuditingEntityListener active).
- Repository injection via `@Autowired`.
- Method names: `givenContext_whenAction_thenExpectedResult()` with `// given` / `// when` / `// then` comments (CLAUDE.md). For exception assertions, use `// when / then` combined with `assertThatThrownBy`.

Required dependencies for test data: `Season`, `Team`, `RaceScoring`, `MatchScoring` are all already populated by `DevDataSeeder` on the `dev` profile, OR they can be created inline in the test using their respective repositories. Use inline creation with a unique test prefix to avoid collisions (e.g. `Season("Phase56-Test", 9999, 1)`, `Team("Phase56-Team-A", "P56A")`). DevDataSeeder seeds many entities — pick simple repositories like `RaceScoringRepository.findAll().get(0)` and `MatchScoringRepository.findAll().get(0)` to fetch existing dev scoring rules without creating new ones.

Constraint-violation pattern (no in-tree precedent for real-DB; derive from AssertJ + Spring's `DataIntegrityViolationException`):

```java
import org.springframework.dao.DataIntegrityViolationException;
import static org.assertj.core.api.Assertions.assertThatThrownBy;

assertThatThrownBy(() -> phaseTeamRepository.saveAndFlush(duplicate))
    .isInstanceOf(DataIntegrityViolationException.class);
```

Use `saveAndFlush` (not `save`) so the constraint violation surfaces inside the test method, not at transaction commit time after the test ends.

Hard rules:
- DO NOT use `@DataJpaTest` (no precedent in the codebase per 56-PATTERNS.md).
- DO NOT create custom repository finders — only `findById`, `findAll`, `save`, `saveAndFlush`, `delete`, `count` are available (D-06).
- DO NOT modify any production code (entities, repositories, migration). This plan is test-only.
- Use AssertJ (`assertThat`, `assertThatThrownBy`) per project convention — JUnit's `Assertions.*` is also acceptable for simple null/equals checks (BaseEntityAuditTest uses JUnit assertions).
