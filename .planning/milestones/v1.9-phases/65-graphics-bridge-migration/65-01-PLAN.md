# Phase 65 — Plan 01: Migrate 5 graphics callers to phase-aware StandingsService API

<objective>
Migrate the five graphics-layer callers of the deprecated `StandingsService.calculateStandings(UUID seasonId)` bridge to the canonical phase-aware `calculateStandings(UUID phaseId, UUID groupId)` API per D-03a..e + D-06 + D-07. The bridge method itself stays intact — its compile-time deletion happens in Plan 65-03. After this plan, the only callers of the bridge are: (a) `StandingsService.calculateBuchholzScores` private helper (cleaned up in Plan 65-02), (b) `SwissPairingService.calculateBuchholz` public method (deleted in Plan 65-02), and (c) the 13 bridge tests in `StandingsServiceTest` (triaged in Plan 65-03).

Purpose: close out 5 of 7 caller sites in one atomic commit so that the LEAGUE-layout regression surface (D-11) and GROUPS-layout-correctness surface (D-12) are explicitly tested at every migrated boundary. After this plan, GROUPS-layout matchday graphics finally show **per-group standings** for the matchday's group instead of falling through the bridge to combined-view standings via `findRegularPhase` (a behavior fix that Plan 65 ships as a side effect of the bridge migration; LEAGUE seasons remain byte-identical).

Output: 5 production files migrated, 5 test files updated (1 newly created — `SettingsGraphicServiceTest.java`), 8 net-new test methods (5 LEAGUE-regression + 3 GROUPS-tests), 7 existing stub rewrites in `AbstractMatchdayGraphicServiceTest`. Single atomic commit `refactor(65): migrate graphics services to phase-aware standings API`.
</objective>
