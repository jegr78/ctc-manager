## Spacing Scale

Declared values (multiples of 4 only). Derived from inline styles in season-detail.html and race-detail.html.

| Token | Value | Usage in this phase |
|-------|-------|---------------------|
| xs | 4px | Icon gaps, badge margin, swatch gap, form-hint margin-top |
| sm | 8px | Compact button padding, gap between inline elements, attachment list gap |
| md | 16px | Default element spacing, card padding override (empty-state--compact), action-bar margin-bottom, mt-md utility |
| lg | 24px | Card internal padding (score-banner), modal-body padding, section spacing |
| xl | 32px | Detail section vertical spacing (mt-lg utility), score-banner gap |
| 2xl | 48px | Full empty-state default padding (existing .empty-state rule) |

Exceptions:
- `gap: 3px` on `.color-swatch-row` — below-scale value retained from existing design for visual density
- `gap: 6px` on `.color-pair` — intentional between-step value for color picker pairs
- `padding: 2px 10px / 2px 8px` on result and context badges — retained from existing badge pattern
- `max-width: 480px / 420px` on modal bodies — kept as single-property inline override per D-04 (not 3+ occurrences)
- `width: 36px / height: 30px` on color picker input — fixed control dimensions, not layout spacing
- `.form-hint { margin-top: 2px }` — retained from existing inline style; hairline spacing below inputs, below minimum scale step
- `.badge--inline { margin-left: 6px }` — retained from existing inline style on badge elements; mid-scale value for tight inline context

Source: CONTEXT.md D-04 (layout utilities only for patterns appearing 3+ times); RESEARCH.md inline style inventory.

**Refactoring Phase Exception — Font Sizes:** This spec documents 9 distinct font sizes in production (11, 12, 13, 14, 16, 20, 22, 24, 36 px). The 4-size constraint from the design contract applies to new design phases where a fresh type scale is being established. This phase is a CSS class extraction: existing inline `font-size` values are being codified into named CSS classes without modification. No new sizes are introduced. The wide range reflects a pre-existing production scale accumulated across multiple development iterations.

**Refactoring Phase Exception — Font Weights:** This spec documents 4 font weights in production (300, 400, 600, 700). The 2-weight constraint applies to new design phases. This phase extracts existing inline `font-weight` declarations into CSS classes without change. All four weights exist in the current production codebase and are retained as-is.

## Color

All colors use CSS custom properties from `:root`. No hardcoded hex values in new CSS classes, with three narrowly-scoped exceptions listed below.

| Role | Value | Usage |
|------|-------|-------|
| Dominant surface (60%) | `--bg: #0a0a0a` | Page background |
| Secondary surface (30%) | `--bg-card: #141414` | Cards, modal body, score banner |
| Input surface | `--bg-input: #1a1a1a` | Form inputs, table rows, sidebar items |
| Accent (10%) | `--accent: #4fc3f7` | Active sidebar links, focus rings, bracket winner, context-badge text |
| Success | `--success-bg / #66bb6a` | WIN result badge background + text, badge-active |
| Danger | `--danger / --danger-bg` | Destructive buttons, error alerts |
| Text primary | `--text: #d0d0d0` | Body copy |
| Text dim | `--text-dim: #999` | Labels, hints, disabled states |
| Text label | `--text-label: #aaa` | Table headers, section labels |
| White | `--white: #fff` | Headings, prominent values |

Hardcoded hex exceptions (retained from existing inline styles, scoped to specific components):
- `color: #66bb6a` in `.result-badge--win` and `.badge-active` — success green not mapped to `--success` variable (existing convention)
- `background: #222` in `.logo-preview-img` — darker-than-card preview background
- `background: #1a2a3a` in `.badge-sub` — existing tinted accent background

Accent reserved for:
- Active sidebar navigation items (border + background tint)
- Focus ring outline on interactive elements
- Bracket winner team highlight
- `.context-badge` text (playoff indicator)
- Quick score button accent tint
- `.border-focus` / input focus ring

Dynamic color values (stay as `th:style` — not replaced by CSS classes):
- Team primary/secondary colors (team color swatches, `color-swatch-row` spans)
- Winner/loser row background coloring in match tables
- Score value color by match result (win = green, loss = red, draw = muted)

Source: CONTEXT.md D-01/D-02; admin.css `:root` block; RESEARCH.md §CSS Custom Properties.

## Visual Verification Protocol

Per D-09 / D-10 and project feedback rule `feedback_playwright_cli.md`:

1. Before any template change: capture baseline screenshot with `playwright-cli screenshot`
2. Dev server: `./mvnw spring-boot:run -Dspring-boot.run.profiles=dev,demo`
3. Screenshot storage: `.screenshots/` directory (never project root)
4. After CSS migration: compare screenshots — pixel-identical result required
5. Any visual difference must be intentional and documented as a PR comment
6. Per-template verification, not per-wave — one screenshot pair per modified template
7. Phase gate: `./mvnw verify -Pe2e` before `/gsd-verify-work`
