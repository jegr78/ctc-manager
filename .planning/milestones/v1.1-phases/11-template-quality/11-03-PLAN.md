**Step 1: Full analysis of template-editors.html.**

Read the entire template-editors.html file. Categorize all 181 `style="..."` occurrences into:
- A) Static styles that map to existing Plan 01 CSS classes (reuse directly)
- B) Static styles that repeat 3+ times within template-editors.html (create new editor-specific classes)
- C) Static styles that appear <3 times (create new class if pattern is clearly semantic, or keep as acceptable single-property inline)
- D) `th:style` / `th:styleappend` attributes (DO NOT TOUCH per D-01)

**Step 2: Define editor-specific CSS classes.**

Based on the analysis, add new classes to admin.css. Expected class categories (based on the editor being a graphic template management UI):
- `.editor-card` — Card container for each graphic template editor
- `.editor-preview` — Preview image/area container
- `.editor-controls` — Control panel with form inputs
- `.editor-field-row` — Row of form fields within editor
- `.editor-field-label` — Label for editor field
- `.editor-color-grid` — Grid of color picker pairs
- `.editor-image-upload` — Image upload area
- `.editor-reset-actions` — Save/reset button row
- Any other patterns that emerge from the 181-style analysis

Name all classes with the `editor-` prefix to scope them to the template-editors page. Follow existing admin.css naming convention (semantic, component-scoped per D-03).

Use CSS custom properties for all colors (`var(--bg-card)`, `var(--border)`, etc.). No hardcoded hex values except the three documented exceptions in UI-SPEC.

**Step 3: Migrate template-editors.html.**

Replace every static `style="..."` with the appropriate CSS class:
- Reuse Plan 01 classes where applicable (`.color-pair`, `.color-picker-input`, `.form-hint`, `.file-input`, `.inline-form`, `.td-center`, etc.)
- Use new editor-specific classes for structural patterns
- Keep `th:style` and `th:styleappend` untouched
- For truly unique single-property styles with <3 occurrences and no semantic grouping, leave as inline (this is acceptable)

**JavaScript check:** template-editors.html likely has significant JavaScript for editor interactions (save, reset, preview, color picking). Scan ALL `<script>` blocks for:
- `element.style.*` assignments — update if the property is now managed by CSS class
- `element.className = '...'` — ensure new CSS classes are included
- `element.classList.add/remove/toggle(...)` — ensure compatibility
Per D-11: when replacing inline styles on JS-manipulated elements, update the JS code to use new CSS classes.

**Size guidance:** With 181 styles, expect ~20-30 new editor-specific classes plus reuse of ~15 classes from Plan 01. The goal is zero static inline styles, but 5-10 acceptable single-property inlines are tolerable for truly unique values.
