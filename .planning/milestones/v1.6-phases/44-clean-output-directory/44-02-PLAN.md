```java
private void cleanOutputDirectory(Path outPath) throws IOException {
    if (!Files.exists(outPath)) {
        return; // D-03: non-existent dir — createDirectories() below handles creation
    }
    log.info("Cleaning output directory: {}", outPath);
    Files.walkFileTree(outPath, new SimpleFileVisitor<>() {
        @Override
        public FileVisitResult visitFile(Path file, BasicFileAttributes attrs) throws IOException {
            Files.delete(file);
            log.debug("Deleted file: {}", file);
            return FileVisitResult.CONTINUE;
        }

        @Override
        public FileVisitResult postVisitDirectory(Path dir, IOException exc) throws IOException {
            if (exc != null) throw exc;
            if (!dir.equals(outPath)) {  // D-02: do not delete root itself
                Files.delete(dir);
                log.debug("Deleted directory: {}", dir);
            }
            return FileVisitResult.CONTINUE;
        }
    });
}
```

Key implementation details per locked decisions:
- D-01: Uses `Files.walkFileTree()` with `SimpleFileVisitor` for bottom-up deletion
- D-02: Root directory preserved (`!dir.equals(outPath)` guard), `Files.createDirectories` recreates it
- D-03: `Files.exists(outPath)` guard returns early for non-existent dirs
- D-04: No path validation — `outputDir` comes from `@Value` property
- Logging: `log.info` for cleanup start (state change), `log.debug` for per-item deletions (Claude's discretion)
- IOException propagation: method declares `throws IOException`, no try/catch wrapping — the outer catch in `generate()` handles it
