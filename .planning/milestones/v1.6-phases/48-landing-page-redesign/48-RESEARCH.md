## Summary

Phase 48 transforms the static site's landing page from a content page (active standings table + last matchday) into a YouTube hero + tile navigation landing page. The work spans four areas: a new `YouTubeScraperService` bean, a complete rewrite of `index.html`, simplification of `generateIndex()`, and a single nav link change in `layout.html`.

All infrastructure is already in place. Jsoup 1.22.1 is a declared dependency, used by `Gt7ScraperService` for live HTTP scraping. `SiteProperties` already uses `@ConfigurationProperties` and is ready for two new fields. The `.overview-card` CSS pattern (Phase 47) provides the tile visual baseline. The only test adaptation needed is removing one test that asserts standings content on the index page and adding ~5 new tests.

The key risk is that YouTube's channel page HTML can change structure, causing the regex to miss `videoId`. The `fallbackVideoId` config field (D-03/D-05) handles this by design: if scraping fails, the configured fallback is returned. Unit tests for `YouTubeScraperService` must therefore mock the Jsoup HTTP call rather than hit the live network.

**Primary recommendation:** Implement `YouTubeScraperService` first (with unit tests), then simplify `generateIndex()`, then rewrite `index.html`, then adjust the nav link, then add integration tests in `SiteGeneratorServiceTest`. This order keeps the compile/test cycle green at each step.

### Pattern 1: Jsoup HTTP Scraping with Fallback

**What:** Fetch a URL, get the raw HTML as string, apply regex to find embedded JSON value.
**When to use:** When the target page embeds data in JavaScript (`ytInitialData` / `videoId` JSON), not in parseable HTML elements.

```java
// Source: Gt7ScraperService pattern (src/main/java/org/ctc/gt7sync/Gt7ScraperService.java)
// Adapted for YouTubeScraperService

public String scrapeVideoId(String channelUrl, String fallbackVideoId) {
    try {
        String html = Jsoup.connect(channelUrl)
                .timeout(10_000)
                .userAgent("Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36")
                .get()
                .html();
        var matcher = Pattern.compile("\"videoId\":\"([a-zA-Z0-9_-]{11})\"").matcher(html);
        if (matcher.find()) {
            return matcher.group(1);
        }
        log.warn("No videoId found on channel page: {}", channelUrl);
    } catch (IOException e) {
        log.warn("Failed to scrape YouTube channel page {}: {}", channelUrl, e.getMessage());
    }
    return fallbackVideoId;
}
```

**Note on Jsoup API:** `Jsoup.connect(url).get()` returns a `Document`; calling `.html()` on it gives the full rendered HTML string including embedded script content. This is the same approach `Gt7ScraperService.fetchText()` uses but with HTML rather than plain text.

### Anti-Patterns to Avoid

- **Hitting live YouTube in unit tests:** `YouTubeScraperService` must use a local mock â€” either inject `Jsoup.Connection` as a strategy or use `@SpyBean` / `PowerMockito`. The cleanest approach (matching project style) is to make the Jsoup call in a package-private method that can be overridden in tests, or inject a `Function<String, String>` for the HTML fetch step. See "Testability" section below.
- **Asserting videoId regex match on real YouTube HTML in integration tests:** YouTube may change its page structure; tests must use fixture HTML, not live network.
- **Leaving old standing/matchday variables in `generateIndex()` context:** Remove them completely. Leaving unused variables in context is dead code and confuses future readers.
- **Setting `currentPage = "index"` (old value):** D-19 requires changing to "home". The existing top-nav active logic `currentPage == 'standings' or currentPage == 'index'` will also need updating â€” "home" must not accidentally activate any top-nav item.

## Testability of YouTubeScraperService

The decision (D-22) requires pure unit tests with mocked Jsoup. `Jsoup.connect()` is a static method â€” it cannot be mocked with standard Mockito without PowerMock.

**Recommended approach (consistent with Gt7ScraperService test pattern):**

Extract the HTTP fetch into a package-private method so tests can subclass or spy:

```java
// In YouTubeScraperService:
// package-private for testing
String fetchChannelHtml(String channelUrl) throws IOException {
    return Jsoup.connect(channelUrl)
            .timeout(10_000)
            .userAgent("Mozilla/5.0 ...")
            .get()
            .html();
}

public String scrapeVideoId(String channelUrl, String fallbackVideoId) {
    try {
        String html = fetchChannelHtml(channelUrl);
        // ... regex
    } catch (IOException e) {
        log.warn(...);
        return fallbackVideoId;
    }
}
```

In the test, use Mockito `spy()`:
```java
// In YouTubeScraperServiceTest:
@ExtendWith(MockitoExtension.class)
class YouTubeScraperServiceTest {

    @Spy
    private YouTubeScraperService service;

    @Test
    void givenValidChannelPage_whenScrapeVideoId_thenReturnsFirstVideoId() throws IOException {
        // given
        doReturn("<html>...\"videoId\":\"dQw4w9WgXcW\"...</html>")
                .when(service).fetchChannelHtml(any());

        // when
        var result = service.scrapeVideoId("https://...", "fallback");

        // then
        assertThat(result).isEqualTo("dQw4w9WgXcW");
    }

    @Test
    void givenIOException_whenScrapeVideoId_thenReturnsFallback() throws IOException {
        // given
        doThrow(new IOException("timeout")).when(service).fetchChannelHtml(any());

        // when
        var result = service.scrapeVideoId("https://...", "fallback-id");

        // then
        assertThat(result).isEqualTo("fallback-id");
    }
}
```

**Alternative (fixture file):** Use test resource files like `Gt7ScraperServiceTest` does. Place a fixture `youtube-channel-stub.html` under `src/test/resources/` and read it in the test, bypassing the HTTP call entirely by calling the regex extraction logic directly (if it is extracted to a separate package-private method).

## Common Pitfalls

### Pitfall 1: currentPage "home" Breaks Existing Nav Active Tests

**What goes wrong:** `givenIndexPage_whenGenerate_thenStandingsTopNavActive` currently expects `Standings` to be active when `currentPage == 'index'`. Changing `currentPage` to "home" and removing "index" from the nav condition causes this test to fail.
**Why it happens:** The nav active class condition is `currentPage == 'standings' or currentPage == 'index'` â€” both must be updated together.
**How to avoid:** Update the layout.html condition AND the test in the same task. The index page intentionally will NOT activate any top-nav item (it is the "home" root, not a content page).
**Warning signs:** Test failure `givenIndexPage_whenGenerate_thenStandingsTopNavActive` after D-17/D-19 changes.

### Pitfall 2: generateIndex() Still Passes standings/matchday Context Variables

**What goes wrong:** If `standings`, `teamSlugMap`, `lastMatchday`, or `lastMatchdayRaces` remain set in the Thymeleaf context, Thymeleaf will not error â€” it simply won't render them (template no longer references them). But the service still executes expensive DB queries for no reason.
**Why it happens:** Partial removal: template updated but service not cleaned up (or vice versa).
**How to avoid:** Remove the entire `if (activeSeason != null)` block that builds standings/matchday data. The `activeSeason` is still needed for the Standings tile URL â€” pass it as a context variable without fetching its standings.
**Warning signs:** Integration tests that verify the index page runs without errors, but `SiteGeneratorService.generate()` takes longer than expected.

### Pitfall 3: Tile Grid 3+2 Layout Fragility on Mobile

**What goes wrong:** `grid-template-columns: repeat(3, 1fr)` with 5 items gives 3+2 on desktop but on narrow screens the 2-item last row is left-aligned, not centered.
**Why it happens:** CSS grid fills left-to-right; the last row has empty cells on the right side.
**How to avoid:** Use `justify-content: center` on the grid plus a fixed tile min-width, or wrap the last 2 tiles in a centered sub-container. The simplest reliable approach is `repeat(auto-fit, minmax(260px, 1fr))` â€” on wide screens this gives 3 tiles per row; on mobile it stacks.
**Warning signs:** Visual inspection with playwright-cli on mobile viewport shows the bottom 2 tiles are left-aligned.

### Pitfall 4: YouTube Scraper Returning Empty fallbackVideoId

**What goes wrong:** If `youtubeVideoId` config is left as empty string and scraping fails (offline, CI, YouTube changes structure), the iframe renders with `src="https://www.youtube.com/embed/"` â€” an invalid URL that shows a blank or broken iframe.
**Why it happens:** The fallback is unconfigured.
**How to avoid:** Either (a) set a real fallback video ID in `application.yml` (and document it), or (b) in the template, add `th:if="${videoId != null and !#strings.isEmpty(videoId)}"` guard on the iframe div so no iframe renders with an empty ID.
**Warning signs:** Generated `index.html` contains `src="https://www.youtube.com/embed/"` with no ID.

### Pitfall 5: "Standings" Nav Link Hidden When No Active Season

**What goes wrong:** D-17 adds `th:if="${activeSeasonSlug != null...}"` to the Standings nav link. This means on a fresh site with no active season, the Standings nav item disappears entirely.
**Why it happens:** This is the intended behavior per D-17.
**How to avoid:** This is by design. The planner should be aware: integration tests that assert Standings appears in the nav will only pass when the test setup creates an active season (which it does in `setUp()`).
**Warning signs:** Test failure if a test generates the site with no active season and asserts on the Standings nav link.

### YouTubeScraperService Skeleton

```java
// [ASSUMED] â€” follows project conventions from CLAUDE.md and Gt7ScraperService
package org.ctc.sitegen;

import lombok.RequiredArgsConstructor;
import lombok.extern.slf4j.Slf4j;
import org.jsoup.Jsoup;
import org.springframework.stereotype.Service;

import java.io.IOException;
import java.util.regex.Pattern;

@Slf4j
@Service
@RequiredArgsConstructor
public class YouTubeScraperService {

    private static final Pattern VIDEO_ID_PATTERN =
            Pattern.compile("\"videoId\":\"([a-zA-Z0-9_-]{11})\"");

    // package-private for testability
    String fetchChannelHtml(String channelUrl) throws IOException {
        return Jsoup.connect(channelUrl)
                .timeout(10_000)
                .userAgent("Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/120.0.0.0 Safari/537.36")
                .get()
                .html();
    }

    public String scrapeVideoId(String channelUrl, String fallbackVideoId) {
        try {
            String html = fetchChannelHtml(channelUrl);
            var matcher = VIDEO_ID_PATTERN.matcher(html);
            if (matcher.find()) {
                String videoId = matcher.group(1);
                log.info("Scraped YouTube videoId: {} from {}", videoId, channelUrl);
                return videoId;
            }
            log.warn("No videoId found on YouTube channel page: {}", channelUrl);
        } catch (IOException e) {
            log.warn("Failed to scrape YouTube channel page {}: {}", channelUrl, e.getMessage());
        }
        return fallbackVideoId;
    }
}
```

## Security Domain

This phase has no user-facing authentication, form inputs, session management, or cryptography. It adds:
- One outbound HTTP call (Jsoup to YouTube) â€” no inbound data from users.
- A static HTML iframe embed â€” no XSS risk (videoId is a server-side-computed value from a trusted regex match on a known URL; it is not user-supplied).
- New config fields â€” no secrets involved.

**ASVS categories applicable:**

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | No | â€” |
| V3 Session Management | No | â€” |
| V4 Access Control | No | â€” |
| V5 Input Validation | Partial | videoId regex constrains to `[a-zA-Z0-9_-]{11}` before embedding in HTML â€” prevents injection |
| V6 Cryptography | No | â€” |

**V5 note:** The videoId extracted by the regex is 11 alphanumeric/hyphen/underscore characters only. This is rendered directly into an iframe `src` attribute. The regex constraint (`[a-zA-Z0-9_-]{11}`) prevents any injection since it excludes all HTML-special characters (`<`, `>`, `"`, `'`, `&`). No additional escaping is needed â€” Thymeleaf's `th:src` also auto-escapes attribute values.
