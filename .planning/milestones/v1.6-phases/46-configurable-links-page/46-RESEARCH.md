# Phase 46: Configurable Links Page - Research

**Researched:** 2026-04-16
**Domain:** Spring Boot `@ConfigurationProperties`, site generator page generation, Thymeleaf templates
**Confidence:** HIGH

## Summary

This phase introduces `SiteProperties` as the project's first `@ConfigurationProperties` class, migrating `ctc.site.output-dir` from `@Value` on `SiteGeneratorService` and adding a `List<LinkEntry> links` field. A new `links.html` page is generated by a `generateLinks()` method in `SiteGeneratorService` that reads the configured links and passes them to a Thymeleaf template following the established `archive.html` pattern.

The implementation is small and well-bounded: one new class (`SiteProperties`), minor surgery to `SiteGeneratorService` (remove two `@Value` fields, inject `SiteProperties`, add one private method and one call site), one new template (`links.html`), CSS additions for link cards, and YAML config additions in two files. All decisions are locked in CONTEXT.md — no alternative exploration required.

**Primary recommendation:** Follow the `generateArchive()` → `archive.html` pattern precisely. `SiteProperties` uses a standard JavaBean-style inner class (not record) because Spring Boot config binding requires a no-arg constructor and setters.

### Pattern 1: @ConfigurationProperties with Nested List

**What:** Type-safe config class bound from YAML using a nested JavaBean inner class for list items.

**When to use:** When multiple related config values share a common prefix — preferred over multiple `@Value` fields.

**Critical constraint:** Spring Boot config binding requires a no-arg constructor + setters on the inner class. Records and Lombok `@Value` do NOT work for nested list items. [VERIFIED: existing codebase uses `@Value` for single values but CONTEXT.md D-03 explicitly mandates JavaBean-style]

```java
// Source: Spring Boot 4.x @ConfigurationProperties binding pattern [ASSUMED from Spring Boot docs pattern]
@ConfigurationProperties(prefix = "ctc.site")
public class SiteProperties {
    private String outputDir;
    private List<LinkEntry> links = List.of();

    // getters + setters for outputDir and links

    public static class LinkEntry {
        private String name;
        private String url;
        // no-arg constructor (implicit)
        // getters + setters for name and url
    }
}
```

YAML binding:
```yaml
ctc:
  site:
    output-dir: docs/site
    links:
      - name: "YouTube"
        url: "https://www.youtube.com/@CommunityTeamCup"
```

**Registration:** Either `@EnableConfigurationProperties(SiteProperties.class)` on a `@Configuration` class, or add `@ConfigurationPropertiesScan` on the main application class. The CONTEXT.md D-04 specifies `@EnableConfigurationProperties` — use a dedicated `@Configuration` class or annotate `SiteGeneratorService` directly. [ASSUMED: both approaches valid in Spring Boot 4.x]

### Anti-Patterns to Avoid

- **Using a record for LinkEntry:** Spring Boot config binding cannot set fields on records via setters. Must use a plain class with no-arg constructor and setters.
- **Calling `List.of()` as the mutable default and then binding to it:** Use `ArrayList` or rely on Spring's own list creation. `List.of()` as a field default is fine — Spring Boot replaces the entire list value during binding, not mutates it.
- **Injecting `outputDir` from both `@Value` and `SiteProperties`:** After migration, `outputDir` comes exclusively from `SiteProperties`. Remove the `@Value("${ctc.site.output-dir}")` field on `SiteGeneratorService` and replace `outputDir` usages with `siteProperties.getOutputDir()`. The `@lombok.Setter` on the field (used by tests via `setOutputDir()`) must be preserved — add a `setOutputDir()` method to `SiteProperties` or keep the setter on the service delegating to the properties object.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Config list binding | Manual YAML parsing | `@ConfigurationProperties` | Spring handles type conversion, validation, relaxed binding |
| HTML template rendering | String concatenation | Thymeleaf (`writeTemplate()`) | Consistent with all other pages; escaping handled |
| External link safety | Custom rel/target logic | HTML attributes `target="_blank" rel="noopener"` | Browser standard |

**Key insight:** The `writeTemplate()` method already handles relative path computation for `assetsPath` and `rootPath`. New `generateLinks()` must use it as-is — no modifications needed.

## Common Pitfalls

### Pitfall 1: Test Setter for outputDir After Migration

**What goes wrong:** The existing test uses `siteGeneratorService.setOutputDir(tempDir.toString())` (line 148 in test). After migrating `outputDir` to `SiteProperties`, this setter disappears from `SiteGeneratorService`.

**Why it happens:** `@lombok.Setter` on the `@Value`-annotated field generates the setter. Once the field is removed, the setter goes with it.

**How to avoid:** Either (a) add a `@Setter` on `outputDir` in `SiteProperties` and expose a `setOutputDir()` delegation method on `SiteGeneratorService`, or (b) add a `@Autowired` `SiteProperties` reference in the test and call `siteProperties.setOutputDir(...)`. Option (a) is simpler and keeps the test unchanged.

**Warning signs:** `NoSuchMethodException` or compilation error on `siteGeneratorService.setOutputDir(...)` in tests.

### Pitfall 2: writeTemplate() Uses `outputDir` Field Directly

**What goes wrong:** `writeTemplate()` at line 418 uses `Path.of(outputDir)` — the field, not a method call. After migration, `outputDir` is no longer a field on `SiteGeneratorService`.

**Why it happens:** `outputDir` field used directly in `writeTemplate()` for relative path computation.

**How to avoid:** Replace `Path.of(outputDir)` with `Path.of(siteProperties.getOutputDir())` in `writeTemplate()`. This is a one-line change.

**Warning signs:** `NullPointerException` or incorrect relative paths in generated HTML.

### Pitfall 3: Empty List Default and YAML Missing Key

**What goes wrong:** If `ctc.site.links` key is absent from YAML entirely, the field must default to an empty list, not null. A `null` list passed to `ctx.setVariable("links", null)` then `${#lists.isEmpty(links)}` in Thymeleaf will throw a `NullPointerException`.

**Why it happens:** `@ConfigurationProperties` leaves unset list fields as their Java default value (null if not initialized).

**How to avoid:** Initialize in `SiteProperties`: `private List<LinkEntry> links = new ArrayList<>()` or `= List.of()`. The `List.of()` default from CONTEXT.md D-02 is safe because Spring Boot replaces, not mutates, the list.

**Warning signs:** NPE during template rendering when `links` is null.

### Pitfall 4: `@EnableConfigurationProperties` Placement

**What goes wrong:** `SiteProperties` is not picked up by Spring if neither `@EnableConfigurationProperties` nor `@ConfigurationPropertiesScan` is present.

**Why it happens:** Unlike `@Component`, `@ConfigurationProperties` alone does not register the bean.

**How to avoid:** Add `@EnableConfigurationProperties(SiteProperties.class)` to `SiteGeneratorService` or a dedicated `@Configuration` class in the `sitegen` package. Since CONTEXT.md D-04 specifies this annotation, prefer a dedicated `SiteGenConfig.java` for clarity.

**Warning signs:** `NoSuchBeanDefinitionException` or config values not binding (fields remain at default).

```java
// Location: src/main/java/org/ctc/sitegen/SiteProperties.java
package org.ctc.sitegen;

import org.springframework.boot.context.properties.ConfigurationProperties;
import java.util.ArrayList;
import java.util.List;

@ConfigurationProperties(prefix = "ctc.site")
public class SiteProperties {

    private String outputDir;
    private List<LinkEntry> links = new ArrayList<>();

    public String getOutputDir() { return outputDir; }
    public void setOutputDir(String outputDir) { this.outputDir = outputDir; }
    public List<LinkEntry> getLinks() { return links; }
    public void setLinks(List<LinkEntry> links) { this.links = links; }

    public static class LinkEntry {
        private String name;
        private String url;

        public String getName() { return name; }
        public void setName(String name) { this.name = name; }
        public String getUrl() { return url; }
        public void setUrl(String url) { this.url = url; }
    }
}
```
