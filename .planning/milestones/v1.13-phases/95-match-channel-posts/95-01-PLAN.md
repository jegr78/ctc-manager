<objective>
Land the POST-01 persistence + service-skeleton + multipart-PATCH webhook scope-add + `/admin/discord/posts` filter listing on top of the Phase-94 channel-lifecycle platform. This plan delivers Flyway V12 (`discord_post`), the `DiscordPost` entity + 12-value `DiscordPostType` enum + repository with `JpaSpecificationExecutor`, the `DiscordPostService.postOrEdit(...)` idempotency pivot, the new `DiscordWebhookClient.editMessageWithAttachments(...)` multipart-PATCH method, and the read-only filter listing page. Plans 95-02..04 wire concrete post types on top of this skeleton.

Per RESEARCH Landmine 1 this plan does NOT bump `BackupSchema.SCHEMA_VERSION` and does NOT add `DiscordPostMixIn` / `DiscordPostRestorer` — `DiscordPost` is structurally excluded from `BackupSchema.EXPORT_ORDER` by the existing `org.ctc.domain.model.*` package filter. Instead a new `DiscordPostGuardTest` mirrors `DiscordGlobalConfigGuardTest` and asserts the invariant (EXPORT_ORDER == 24, SCHEMA_VERSION == 1, no `discord_post` row in EXPORT_ORDER).
</objective>

<must_haves>
- **Truths (observable):**
  - Operator can navigate to `/admin/discord/posts`, see an (initially empty) paginated table with filter inputs for season, match, and post-type, and submit the filter form.
  - When a `discord_post` row exists for a `(channel_id, post_type, match_id)` tuple, `DiscordPostService.postOrEdit(...)` routes to Webhook-PATCH (no duplicate Discord message); when no row exists it routes to Webhook-POST.
  - When called with `List<NamedAttachment>` of size > 0, `postOrEdit` uses the new `DiscordWebhookClient.editMessageWithAttachments(...)` multipart-PATCH; when called empty (SCHEDULE case) it uses the existing `editMessage(...)` JSON-PATCH.
