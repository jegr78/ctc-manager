<objective>
Wire the POST-03 Settings + Lineups multipart-bundle posts on top of Plans 95-01 + 95-02. One multipart-POST per type with N attachments (one PNG per race; race ordering via list-index over `match.getRaces()` per RESEARCH Landmine 2 — there is no `raceNumber` field). Pre-flight strict per D-95-03b: buttons visible only when all races are complete (SETTINGS: all `race.getSettings() != null`; LINEUPS: all races have a non-empty `RaceLineup`); a 5th error category `data-incomplete` (rendered via new `.error-badge--data-incomplete` CSS variant) handles the post-attempt-when-incomplete path.

Per RESEARCH Pitfall 7 the multipart attachments are loaded via `DiscordPostService.readPng` (path-traversal guarded, introduced in Plan 95-02). Per RESEARCH Pitfall 6 the pre-flight predicate for LINEUPS does an N+1 lookup against `RaceLineupRepository.findByRaceId` per race (acceptable for 3-4 races per match).

`BusinessRuleException` from `org.ctc.domain.exception` is REUSED (it already exists and is thrown by `DiscordChannelService.assertPreconditions` line 102-113 for the same pre-flight pattern; `MatchController.createDiscordChannel` already catches it via `applyErrorFlash(ra, e, "Create Discord Channel")`). This plan does NOT create a duplicate class — the executor imports the existing one.
</objective>

<must_haves>
- **Truths (observable):**
  - When operator clicks "Post Settings" on a match where all races have settings configured, ONE multipart-POST lands in Discord containing N PNG attachments named `settings-race-1.png`, `settings-race-2.png`, … in chronological dateTime order (per RESEARCH Landmine 2 + `@OrderBy("dateTime ASC NULLS LAST")` on `Match.races`).
  - When operator clicks "Post Lineups" on a match where all races have a `RaceLineup` with ≥1 driver per team, ONE multipart-POST lands with N PNGs named `lineups-race-{i+1}.png`.
  - Re-Post for SETTINGS or LINEUPS edits the existing Discord message via multipart-PATCH (no duplicate); `discord_post.attachments_replaced_at` advances.
  - When a match has incomplete settings (any race has `settings == null`), the SETTINGS button is rendered as a DISABLED `<span class="btn btn-secondary btn-sm disabled">Post Settings</span>` with tooltip `"Configure settings for all races first"` — no form, no click target.
  - When operator nevertheless triggers the POST endpoint directly (e.g. URL manipulation), `DiscordPostService.postSettings` throws `BusinessRuleException("Configure settings for all races first")` (the existing class from `org.ctc.domain.exception`), the controller catches it and flashes `errorMessage="…"` + `errorCategory="data-incomplete"` (5th category).
  - The `.error-badge--data-incomplete` CSS variant renders a yellow/orange badge distinct from `transient`/`auth`/`not-found`/`category-full`.
- **Artifacts (must exist):**
  - `DiscordPostService.postSettings(Match)` — assembles multipart bundle from `match.getRaces()` (list-index order), each race PNG via `settingsGraphicService.generateSettings(race)` URL → `readPng(url)` byte[].
  - `DiscordPostService.postLineups(Match)` — analog using `lineupGraphicService.generateLineup(race)`.
  - `DiscordPostService.matchHasCompleteSettings(Match)` returns `!races.isEmpty() && races.stream().allMatch(r -> r.getSettings() != null)`.
  - `DiscordPostService.matchHasCompleteLineups(Match)` returns `!races.isEmpty() && races.stream().allMatch(r -> !raceLineupRepository.findByRaceId(r.getId()).isEmpty())`.
  - `DiscordPostService` imports `org.ctc.domain.exception.BusinessRuleException` — no new class is created under `org.ctc.discord.exception` (the existing class is reused; `DiscordChannelService.assertPreconditions` already throws it for the same pre-flight pattern).
  - `MatchController.detail` GET method adds model attributes `settingsPost`, `lineupsPost`, `matchHasCompleteSettings`, `matchHasCompleteLineups` (booleans) so the template can render visibility + disabled-tooltip states without business logic.
  - `MatchController.applyErrorFlash` already accepts `BusinessRuleException e` (current production behavior — see line 132/158 of `MatchController`); the new endpoints reuse the same helper. NO duplicate `applyBusinessRuleFlash` helper is introduced — the existing `applyErrorFlash(BusinessRuleException, String)` overload is the single route for this category. The controller maps the category label to `errorCategory="data-incomplete"` when the caught exception is `BusinessRuleException` (extend the existing helper if it currently sets a different category label).
  - `admin.css` contains a new `.error-badge--data-incomplete` class with distinct color from the 4 existing variants.
- **Key links (critical connections):**
  - `MatchController.detail` GET → preloads `settingsPost` + `lineupsPost` from `discordPostRepository.findByChannelIdAndPostTypeAndMatchId(...)`.
  - `MatchController.detail` GET → preloads `matchHasCompleteSettings` + `matchHasCompleteLineups` booleans via `discordPostService.matchHasCompleteSettings(match)` + `matchHasCompleteLineups(match)`.
  - `MatchController.postSettings` POST endpoint → `discordPostService.postSettings(match)` (which calls `postOrEdit` with N attachments).
  - `DiscordPostService.postSettings` raises `org.ctc.domain.exception.BusinessRuleException` if pre-flight fails (defensive — buttons are gated, but URL manipulation bypasses the UI).
  - `match-detail.html` `.discord-actions--posts` block contains the Settings + Lineups blocks per RESEARCH Pattern + PATTERNS § "match-detail.html".
</must_haves>
