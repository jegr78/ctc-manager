## Integration Warnings (non-blocking)

### Warning 1 — UX-01 scope gap on CsvImportController

`CsvImportController` (race-results sheet-import, lines 115 + 204) still catches typed `GoogleApiException` subtypes as plain `IOException` and emits the generic `Error reading Google Sheet: <e.getMessage()>` flash. The same `GoogleSheetsService` API that Plan 91-02 made type-safe is now consumed by 3 controllers, but only 2 (`DriverSheetImportController` + `RaceController`) got the typed-catch + `errorCategory` + badge UX treatment.

**Test evidence**: `CsvImportControllerExceptionTest.java:66-84` actively confirms the asymmetry — stubs `TransientGoogleApiException` throw, asserts only `errorMessage` exists in model, does NOT assert `errorCategory`.

**Threat re-opened**: T-91-02-IL info-leak (`e.getMessage()` echoed to flash) was explicitly closed in the 2 controllers Plan 91-02 named — but is re-introduced in CsvImportController for the same upstream exception type.

**Recommendation**: v1.13 follow-up plan applies typed-catch + `errorCategory` flash + badge UX to `CsvImportController` for parity.

### Warning 2 — CLEAN-02 grep-predicate drift

REQUIREMENTS.md defines CLEAN-02 verification as `grep -rn "Assumptions\." src/test/java | wc -l = 0`. Phase 88 VERIFICATION.md recorded this as `✓ PASS` at 2026-05-19T08:15:18Z. Phase 89 (PERF-01) then added `BackupStagingDirPerForkIT.java:12,37` using AssertJ's `Assumptions.assumeThat(forkNum).isNotBlank()` — different package (`org.assertj.core.api.Assumptions` vs `org.junit.jupiter.api.Assumptions`), different intent (non-vacuous guard vs Windows-conditional skip), but the grep predicate cannot distinguish.

Current `grep -rn "Assumptions\." src/test/java | wc -l` returns **1** (the legitimate Phase 89 use).

**Recommendation**: v1.13 tighten the CLEAN-02 grep to `grep -rn "org\.junit\.jupiter\.api\.Assumptions"` (JUnit only) OR add `BackupStagingDirPerForkIT.java` to a whitelist documented in REQUIREMENTS.md.

### Warning 3 — Squash-merge subject discipline not codified in runbook

`.github/workflows/release.yml:70,85` requires the squash-merge commit subject to match `^feat(\(.+\))?:` for the v1.12 → v1.12.0 minor bump:

| Operator action | Result |
|-----------------|--------|
| `gh pr merge --squash --subject "feat(v1.12): driver-import gap-closure & test performance round 2"` | ✓ v1.12.0 minor bump |
| `gh pr merge --squash` (no `--subject`) | ⚠ default subject = PR title → if PR title lacks Conventional prefix, workflow silently skips |
| `gh pr merge --squash --subject "release v1.12"` (no prefix) | ⚠ workflow silently skips |
| `gh pr merge --squash --subject "fix: ..."` | ⚠ v1.11.1 published (PATCH instead of MINOR) |

Memory `[[feedback-squash-merge-message]]` records this lesson but the runbook `§ 6 Future-proof releases` does NOT spell it out.

**Recommendation**: Add a one-line `**Squash-merge subject MUST start with \`feat(vX.Y):\` for the MINOR bump to vX.Y.0**` note to `docs/operations/release-runbook.md § 6`. Optional: add a pre-merge checklist item to the PR description for the same.
