# Backup & Restore

> Available since v1.10. Accessible at `/admin/backup`.

## Overview

CTC Manager v1.10 ships a full database backup/restore feature for admins. A backup captures all
24 operative entity tables (seasons, matchdays, races, results, drivers, teams, etc.) and the
`uploads/` tree (team logos, race attachments) into a single ZIP. An import replaces the live
database atomically — schema-version-gated to prevent catastrophic data loss across major versions.

The feature lives at `/admin/backup`. Every import is preceded by an automatic backup-before-import
ZIP stored at `data/.import-backups/<timestamp>/` for 24-hour manual recovery.

## Export

![Backup page](https://raw.githubusercontent.com/jegr78/ctc-manager/master/.screenshots/77/01-backup-page.png)

1. Sign in to the admin area and click **Backup** in the sidebar — you land on `/admin/backup`.
2. Click **Export Backup**. A ZIP file (`ctc-backup-<ISO-instant>.zip`) is streamed to your browser
   immediately — no full-dataset buffering, so very large leagues download progressively.
3. Store the ZIP in a safe location. The ZIP layout is:
   - `manifest.json` (first entry) — schema version + per-table row counts + export timestamp.
   - `data/<entity>.json` files — one per entity, JSON-serialized via the dedicated `backupObjectMapper`.
   - `uploads/` — mirror of files actually referenced by entities (logos, attachments).

## Import

![Preview screen](https://raw.githubusercontent.com/jegr78/ctc-manager/master/.screenshots/77/02-preview-screen.png)

1. On `/admin/backup`, choose your backup ZIP via **Import Backup** and submit.
2. Review the preview screen: per-table row counts (current vs. backup), the schema-version match
   indicator, and the upload-file count. The database is NOT touched until you confirm.
3. Check the **I understand** confirmation and click **Execute Import**. The entire database is
   wiped and replaced inside a single transaction. The previous `uploads/` tree is preserved at
   `data/.import-backups/<timestamp>/uploads-old/` for 24-hour manual recovery.

While an import is in flight, every admin page shows a yellow read-only banner; any write attempt
on another admin page is rejected with HTTP 503 until the import completes.

![Read-only banner during import](https://raw.githubusercontent.com/jegr78/ctc-manager/master/.screenshots/77/03-import-banner.png)

## Schema Version

The backup format carries a `schema_version` integer (currently `1`). If you upload a backup whose
schema version does NOT match the running application's `SCHEMA_VERSION`, the import is rejected
with HTTP 400 BEFORE any database write — no rollback necessary, the database is untouched.

The schema version is bumped manually when the on-disk wire shape changes incompatibly (added /
removed / renamed entity fields, FK changes). The strict-equality check is a deliberate
catastrophic-data-loss guard for v1.x; cross-version migration is a future feature.

## Recovery

If an import fails or you need to revert to the pre-import state:

1. The pre-import automatic backup is at `data/.import-backups/<timestamp>/auto-backup-before-import.zip`.
2. The previous `uploads/` tree is at `data/.import-backups/<timestamp>/uploads-old/`.
3. Both are retained for 24 hours. Step-by-step recovery is documented in the operator runbook:

[`docs/operations/import-runbook.md`](https://github.com/jegr78/ctc-manager/blob/master/docs/operations/import-runbook.md)

The runbook covers: re-importing the pre-import auto-backup, manually restoring the `uploads-old/`
tree, and verifying integrity via the `data_import_audit` table.

## Screenshots

All screenshots are stored in the main repository under `.screenshots/77/` and embedded here via
stable `raw.githubusercontent.com/jegr78/ctc-manager/master/` URLs:

- [01-backup-page.png](https://raw.githubusercontent.com/jegr78/ctc-manager/master/.screenshots/77/01-backup-page.png)
- [02-preview-screen.png](https://raw.githubusercontent.com/jegr78/ctc-manager/master/.screenshots/77/02-preview-screen.png)

The wiki repo does NOT carry binary screenshot copies — the URLs point at the main repo's master
branch, so when screenshots are re-captured in a future phase, this wiki page auto-updates without
any wiki-side change.
