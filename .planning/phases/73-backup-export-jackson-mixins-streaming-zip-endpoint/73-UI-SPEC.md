# Phase 73 — UI Design Contract

> Visual and interaction contract for the `/admin/backup` page introduced in Phase 73 (Backup Export — Jackson MixIns + Streaming ZIP Endpoint).
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope Recap (informative)

Phase 73 ships **exactly one new user-facing page** and **one new sidebar link**:

- **Page**: `GET /admin/backup` — server-rendered Thymeleaf template at `src/main/resources/templates/admin/backup.html`.
- **Action**: a single primary button **Export Backup** that submits a CSRF-protected POST form to `/admin/backup/export`. The response is a `StreamingResponseBody` ZIP download (`Content-Disposition: attachment; filename=ctc-backup-<ISO-instant>.zip`).
- **Sidebar**: a new top-level sidebar group **Data** containing a single entry **Backup** (positioned between the existing `Tools` group and the `Generate Site` link).

Out of scope for this contract (deferred to Phase 74+): import-preview UI, progress bar, history list, retry/cancel affordances.

This contract is intentionally minimal and re-uses the existing `admin.css` system. No new CSS classes, no new fonts, no new colors, no inline styles. The closest live analog is `src/main/resources/templates/admin/generate.html` — Phase 73 mirrors that page's structure.

**60/30/10 split (concrete percentages on the rendered Phase 73 page):**

- 60% — `--bg` (`#0a0a0a`) covers the body and the area outside the card. Roughly 60% of the viewport on a 1440×900 desktop.
- 30% — `--bg-card` (`#141414`) covers the single `.card` block and `#111` covers the 220px sidebar. Together ~30% of the viewport.
- 10% — Accent `#4fc3f7` appears **only** on (a) the cyan `border-left` of the active `Backup` sidebar entry (3px stripe, admin.css line 116-119), (b) focus outlines on the button (`outline: 2px solid var(--border-focus)`, admin.css line 229-232), and (c) the cyan-tinted background `rgba(79, 195, 247, 0.08)` of the active sidebar row. Total: ≤10% of pixels.

**Accent reserved for (explicit list — Phase 73 may NOT broaden this):**

1. The cyan `border-left: 3px solid var(--accent)` on the active sidebar entry (the **Backup** link when the user is on `/admin/backup`).
2. The cyan focus-visible outline on the Export Backup button (`btn:focus-visible`).
3. The cyan focus-visible outline on the sidebar `Backup` link (`sidebar a:focus-visible` if present in CSS; otherwise via global focus styles).

The Export Backup button itself is **NOT** accent-colored. It uses `.btn-primary` (blue `#1976d2`, admin.css line 234) — the established primary-action color used by every other primary submit button in the admin (Preview, Save, Create, Generate Site uses `.btn-success`).

**Why `.btn-primary` (blue) and not `.btn-success` (green) for Export Backup?** `.btn-success` is reserved by convention for `Generate Site` (admin.css line 121-135) — a site-build action with the same green visual identity as the green sidebar pill. Phase 73's export is not a "publish to production" action; it is a read-only data export. `.btn-primary` (blue) is the right semantic match — it lines up with `Preview` on `/admin/import` (import.html line 96), which is the closest peer action. `.btn-danger` (red) is reserved for destructive actions (delete operations); export is non-destructive, so red is wrong.

## Layout (Page Structure)

The page renders inside the standard `admin/layout :: layout(title, content)` Thymeleaf fragment (admin/layout.html line 2). The layout already provides:

- Sidebar (220px fixed left, admin.css line 51-62)
- Main content area (max-width 1200px, container padding 24/32, admin.css line 138-147)
- Flash message slots (`.alert-success`, `.alert-error` — admin.css line 153-163; rendered by `layout.html` line 78-79 automatically from `successMessage` / `errorMessage` flash attributes)

**Page structure for `backup.html` (mirrors `generate.html` exactly — see admin/generate.html):**

```
<section>
    <h1>Backup</h1>              <!-- 22px Conthrax, white, mb-md (16px) -->
    <div class="card">           <!-- 24px padding, bg-card, border, radius-md -->
        <p class="text-dim mb-md">
            Exports the full league database (all 23 entities) plus every uploaded file
            into a single ZIP archive. The download starts immediately and may take a few
            seconds for large leagues.
        </p>
        <form th:action="@{/admin/backup/export}" method="post">
            <button type="submit" class="btn btn-primary btn-lg">
                Export Backup
            </button>
        </form>
    </div>
</section>
```

**Notes for the executor:**

- No inline styles on the button or anywhere else. Mandatory CLAUDE.md rule.
- The `th:action="@{/admin/backup/export}"` with `method="post"` triggers Thymeleaf's automatic CSRF token injection (Spring Security default behavior in prod/docker — verified against admin/generate.html which uses the same pattern without an explicit hidden field). On dev/local the `OpenSecurityConfig` disables CSRF, so the same template works in all profiles.
- The page has **no JavaScript**. No `<script>` block. Click → form POST → browser handles the streamed download via `Content-Disposition`.
- The button is rendered un-disabled. After click, the browser navigates the form submission; the server responds with the streaming ZIP. There is no spinner or "exporting…" disabled-state UI in Phase 73 (the export typically completes within a few seconds; UX-complexity of streaming-state UI is explicitly out of scope per the orchestrator brief).

**Page header pattern:** uses the `<h1>` directly inside `<section>` — same as every other admin page (seasons.html, teams.html, drivers.html, generate.html, import.html). No breadcrumb, no toolbar — the page has only one action.

## Sidebar Entry

A new sidebar group **Data** is added to `admin/layout.html` between the existing `Tools` group (which closes at `</div>` after the `Generate Site` link, layout.html line 73) and the closing `</nav>`. The new fragment:

```html
<div class="sidebar-group">
    <span class="sidebar-group-label">Data</span>
    <a th:href="@{/admin/backup}" th:classappend="${title.contains('Backup') ? 'active' : ''}">Backup</a>
</div>
```

**Rationale for a new group (vs. adding to `Tools`):**

- `Tools` currently contains `Standings`, `Power Rankings`, `Import` (driver-sheet preview), `GT7 Sync` (master-data scraping), `Team Cards`, `Template Editors`, `Generate Site`. These are **per-feature operational tools**.
- Backup is a **whole-database lifecycle action**, mentally distinct. Putting it in its own group reserves slot space for the upcoming Phase 74 `Backup Import / Restore` link without re-shuffling `Tools`.
- The `Data` group label is rendered in the same `.sidebar-group-label` style (11px, uppercase, `#666`, letter-spacing 1px — admin.css line 91-98) as every other group label.

**Active-state detection:** `th:classappend="${title.contains('Backup') ? 'active' : ''}"` follows the same pattern as the other 17 sidebar entries (e.g. `Seasons`, `Teams`, `Drivers` — all use `title.contains('…')` against the controller-supplied `pageTitle`). The controller for `GET /admin/backup` must therefore set `model.addAttribute("pageTitle", "Backup")` (or pass `"Backup"` as the layout title argument — both wire to the same `title` variable on layout.html line 2).

**Mobile behavior:** the sidebar collapses behind the hamburger toggle button (admin.css line 31-35 + `js/sidebar-toggle.js`). No Phase 73-specific changes — the new `Data` group inherits the existing mobile pattern automatically.

## Interaction Contract

| Trigger | Effect | Failure mode |
|---------|--------|--------------|
| User clicks **Backup** in the sidebar | Browser navigates `GET /admin/backup`. Controller adds `title="Backup"` to model and returns `"admin/backup"`. Layout renders, sidebar entry shows `.active` cyan border-left. | If controller missing: 404 via `GlobalExceptionHandler` → renders `admin/error.html`. |
| User clicks **Export Backup** | Browser submits the form (POST `/admin/backup/export`) with the auto-injected `_csrf` hidden field (prod/docker) or no token (dev/local where CSRF is disabled). Controller method returns a `StreamingResponseBody`; Spring sets `Content-Disposition: attachment; filename=ctc-backup-<ISO-instant>.zip` and streams the ZIP. The browser shows a file-save dialog or auto-downloads (depending on user settings). The current page remains visible — there is no redirect, no spinner. | If `BackupExportService` throws before the first byte is flushed: controller catches, sets `errorMessage` flash, redirects to `/admin/backup`. If the exception occurs mid-stream (after `Content-Disposition` headers are already sent): the ZIP truncates; browser shows a partial download. Phase 73 logs the stack trace via `@Slf4j log.error()` but cannot surface a UI message because the response is already committed. (This matches the orchestrator brief: "Phase 73 is fire-and-forget download.") |
| Anonymous request on prod/docker | Spring Security returns 401 + Basic Auth challenge. | n/a — handled by framework. |
| Anonymous request on dev/local | Request succeeds (no auth required in `OpenSecurityConfig`). Same export flow as authenticated. | n/a — by design. |
| Concurrent click (user clicks Export Backup twice quickly) | Two parallel export streams. Phase 73 accepts this; it is a `@Transactional(readOnly=true)` operation with no shared mutable state. **Concurrent-export coordination is Phase 76 (SECU-05 import-lock — and even there the lock applies to imports, not exports).** | No corruption; both downloads complete independently. |

**Disabled-while-streaming behavior:** **none in Phase 73.** The button is not disabled after click. Justification: (a) ZIP download is typically <5 seconds for current dataset sizes (no measured pain point); (b) adding JS to disable the button introduces a new client-side state-machine that conflicts with the project's "no frontend build, no JS framework" principle (STACK.md line 26); (c) the orchestrator brief explicitly lists "Progress bar / streaming status UI" as out of scope. If, after live UAT, this proves to be a footgun, it can be added in a follow-up phase with a single ~5-line vanilla `onclick="this.disabled=true; setTimeout(()=>this.disabled=false, 10000); this.form.submit();"` and a CSS-only "Exporting…" label swap.

## Accessibility

Inherited from the existing admin layout; no new accessibility commitments needed for this lean page. Specific guarantees:

- **Keyboard navigation:** Tab order reaches the sidebar `Backup` link, then the page H1 (non-interactive but discoverable), then the **Export Backup** submit button. All three render the cyan `.focus-visible` outline (admin.css line 229-232 — `outline: 2px solid var(--border-focus)`).
- **Color contrast:** `#1976d2` (button background) against white text → 4.57:1 (WCAG AA pass). `#d0d0d0` body text against `#0a0a0a` background → 14.4:1 (WCAG AAA pass). `#999` `.text-dim` against `#0a0a0a` → 5.93:1 (WCAG AA pass). The button label uses white-on-blue which is the strongest contrast pair in the admin theme.
- **Screen reader semantics:** `<button type="submit">Export Backup</button>` inside a labeled `<form>` is announced as "Export Backup, button, submit form". No ARIA additions needed.
- **No motion:** the page has no animations, no transitions other than the existing `.btn` `transition: all 0.2s` hover (admin.css line 226). Respects `prefers-reduced-motion` by virtue of not animating anything substantive.
- **Touch target:** the `.btn .btn-lg` is 16px font with 12/24 padding → effective height ≥ 44px (meets WCAG 2.5.5 AAA recommendation for touch targets).
