<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| PR diff → `changes` job classification | The `code`/`docs` decision determines whether tests/build run; a misclassification can bypass the gate |
| GHA cache backend → Docker build | Restored cache layers are reused in the image build |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-106-01 | Tampering (CI-gate bypass) | `changes` job `code` filter | mitigate | The `code` filter is a SAFE SUPERSET: it includes `src/**`, `pom.xml`, `Dockerfile`, `.github/workflows/**`, `mvnw`, `.mvn/**` (D-05). Anything not provably docs falls through to `code='true'` → full build runs. A code-changing PR cannot be classified docs-only because every build-relevant tree is enumerated in `code`. The `docs` filter is informational only and never short-circuits the build. |
| T-106-02 | Elevation of Privilege | fork-PR access to changed-file list | accept | `changes` job uses `permissions: pull-requests: read` only; no secrets exposed. dorny/paths-filter falls back to the GitHub API for fork PRs with read-only scope. |
| T-106-03 | Spoofing/Tampering | Docker GHA cache poisoning across scopes | mitigate | `scope=ctc-docker` isolates the cache key from `release.yml` and from `actions/cache` (Maven/Playwright) namespaces; cache restore cannot inject foreign layers into this build. |
| T-106-SC | Tampering | new action references (paths-filter, buildx, build-push) | accept | All three are RESEARCH-VERIFIED against official release pages, pinned to exact versions (no floating tags); no npm/pip/cargo installs in this plan. |
</threat_model>
