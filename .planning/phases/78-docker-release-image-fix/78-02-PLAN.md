<objective>
Add two structural protections to `.github/workflows/ci.yml` so the Phase 78 Dockerfile pin cannot silently regress:

1. A **build-guard step** that scans `Dockerfile` for `FROM eclipse-temurin:` lines and fails the job if any such line does not end in `-noble`. Whitelist-on-suffix approach (D-05), opinionated failure message naming Phase 78 (D-04 wording style), cross-platform `grep -F` / `grep -E` idiom mirroring Phase 71-05 (commit `f451ff4`).
2. A **`docker build .` job** that exercises both Dockerfile stages — including the `RUN ... playwright install chromium` step inside stage 2 — on every PR and every push to master (D-06, D-07, D-08). The point is that this CI job would have caught release run `25609204039` on PR instead of on release.

Purpose: Belt-and-suspenders defense. Plan 01's inline comments warn the next maintainer; this plan structurally enforces the pin so a contributor cannot merge an unpinned Dockerfile even if they ignore or remove the comments.
Output: Two additions to `.github/workflows/ci.yml`: (a) a fast `dockerfile-noble-pin-guard` job (independent of the heavy build-and-test and docker-build jobs, so it fails in &lt; 10 seconds on regressions); (b) a `docker-build` job that performs `docker build .` and emits a tagged local image (no registry push, no login).
</objective>

    ```yaml
      dockerfile-noble-pin-guard:
        # Phase 78: Structural guard for the eclipse-temurin -noble suffix pin in Dockerfile.
        # Fails if any `FROM eclipse-temurin:` line does not end in `-noble`.
        # Whitelist-on-suffix approach (D-05). Failure message references Phase 78 (D-04 style).
        # Cross-platform grep idiom mirrors pom.xml PLAT-07 build-guard (commit f451ff4).
        runs-on: ubuntu-latest

        steps:
          - name: Checkout
            uses: actions/checkout@v6

          - name: Dockerfile -noble pin guard
            run: |
              set -euo pipefail
              # Extract every FROM eclipse-temurin: line (anchored, ignoring trailing whitespace tolerated by Docker).
              from_lines=$(grep -nE '^FROM eclipse-temurin:' Dockerfile || true)
              if [ -z "$from_lines" ]; then
                echo "[Phase 78 build-guard] ERROR: No 'FROM eclipse-temurin:' lines found in Dockerfile."
                echo "Either the base image was changed away from eclipse-temurin (update this guard) or the Dockerfile is missing."
                exit 1
              fi
              # Whitelist: every FROM eclipse-temurin: line MUST contain '-noble' before end-of-line or ' AS '.
              # grep -vE removes whitelisted lines (lines that DO contain -noble in the tag).
              # grep -vF is reserved for fixed-string filters per the Phase 71-05 portability pattern.
              violations=$(printf '%s\n' "$from_lines" | grep -vE ':[^[:space:]]*-noble([[:space:]]|$)' || true)
              if [ -n "$violations" ]; then
                echo "[Phase 78 build-guard] FAIL: Dockerfile uses an unpinned eclipse-temurin tag."
                echo "All 'FROM eclipse-temurin:' lines must end in '-noble' (e.g. '25-jdk-noble', '25-jre-noble')."
                echo "Reason: Playwright 1.59.0 does not support Ubuntu 26.04 (Plucky), which the bare '25-jre' tag silently rotated to in release run 25609204039."
                echo "See .planning/phases/78-docker-release-image-fix/78-CONTEXT.md (decisions D-01, D-05, D-06)."
                echo ""
                echo "Offending line(s):"
                echo "$violations"
                exit 1
              fi
              echo "[Phase 78 build-guard] OK - all 'FROM eclipse-temurin:' lines are pinned to -noble."
              echo "$from_lines"
    ```

    Implementation notes for the executor:

    1. The guard runs ON every PR and every push to master because it inherits the workflow-level `on:` trigger (lines 3-7 of the file, which already covers `push: branches: [master, main]` and `pull_request: branches: [master, main]`). Do NOT add a per-job `on:` filter — D-06 requires structural-not-advisory, so no path filtering, no `if:` conditions.

    2. The guard regex `:[^[:space:]]*-noble([[:space:]]|$)` matches the tag portion (everything after `eclipse-temurin:` up to whitespace or end-of-line) and requires `-noble` followed by whitespace (the ` AS build` case for stage 1) or end-of-line (the stage 2 case). This correctly accepts both `eclipse-temurin:25-jdk-noble AS build` and `eclipse-temurin:25-jre-noble`, and correctly rejects `eclipse-temurin:25-jdk` and `eclipse-temurin:25-jre`.

    3. Use `set -euo pipefail` so a shell error or unset variable fails the job immediately (the Phase 71-05 idiom uses bare bash, but Phase 78 prefers strict-mode because the guard runs on every PR and a silent skip would defeat the purpose).

    4. Cross-platform grep: the script runs on `ubuntu-latest` (GNU grep), so `grep -E` / `grep -vE` is fine in practice. The reason to mirror Phase 71-05's `grep -F` idiom is consistency with the project's established build-guard pattern — if the runner ever changes to a BSD-grep host, the script still works.

    5. The failure message wording matches the &lt;specifics&gt; section of 78-CONTEXT.md: it names Phase 78 explicitly, names `-noble`, points to 78-CONTEXT.md, and uses the `[Phase 78 build-guard]` tag for grep-ability in CI logs.

    6. Do NOT introduce any `if:` filters on the workflow trigger. The user explicitly accepted "+1-3 min CI per PR" (D-07) so the guard runs always.

      docker-build:
        # Phase 78: Exercises `docker build .` on every PR + push to master so future
        # Dockerfile / base-image regressions fail fast on PR instead of on release.
        # In particular, exercises stage 2's `playwright install chromium` RUN step —

        # Acceptable cost: +1-3 minutes CI per PR (explicitly accepted by user).

        runs-on: ubuntu-latest

        steps:
          - name: Checkout
            uses: actions/checkout@v6

          - name: Build Docker image (full multi-stage, no push)
            id: docker_build
            run: |
              set -euo pipefail
              IMAGE_TAG="ctc-manager:ci-${GITHUB_SHA}"
              echo "Building $IMAGE_TAG from pinned -noble base..."
              docker build -t "$IMAGE_TAG" . 2>&1 | tee /tmp/docker-build.log
              echo "Built $IMAGE_TAG successfully."
              docker image inspect "$IMAGE_TAG" --format 'Image ID: {{.Id}} Size: {{.Size}}'

          - name: Upload docker build log on failure
            if: failure()
            uses: actions/upload-artifact@v7
            with:
              name: docker-build-log
              path: /tmp/docker-build.log
              retention-days: 7

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Contributor PR → ci.yml | A contributor could remove or weaken the guard step. Detection: the guard removal is itself a visible diff under `.github/workflows/`, reviewable in code review. No structural enforcement beyond review (acceptable for a single-admin repo). |
| Contributor PR → Dockerfile | A contributor could replace `-noble` with another tag. Detection: the guard fails the PR before merge. This is the primary mitigation. |
| GitHub Actions runner → Docker Hub | The CI docker-build job pulls `eclipse-temurin:25-{jdk,jre}-noble`. If Docker Hub serves a maliciously tampered Noble image, the build could produce a compromised local image. Out of scope (no SHA256 digest pin per D-01; revisit if a real incident occurs). |
