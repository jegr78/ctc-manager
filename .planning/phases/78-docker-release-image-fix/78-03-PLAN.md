<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Developer laptop docker daemon → upstream Docker Hub | Pulls `eclipse-temurin:25-{jdk,jre}-noble` and `mvnw`-downloaded Maven dependencies. Trust = upstream registry integrity. Same trust surface as CI; no incremental risk. |
| Local network → container port 18080 | Container is bound to localhost only (`-p 18080:8080`, not `-p 0.0.0.0:18080:8080`). No external-network exposure. |
| Developer keystrokes → resume signal text | Pasting the curl response is voluntary disclosure of routine health metadata (`{"status":"UP"}`). No secrets present. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-78-09 | Tampering (local image accept) | Built `ctc-manager:phase78-local` image | mitigate | The two-step verification (build + health) gives "watched it work" signal. The `docker stop` + `--rm` ensures the smoke container is cleaned up after the check. |
| T-78-10 | Information Disclosure | `/tmp/phase78-*.log` artifacts | accept | Logs contain Spring Boot startup output (resolved pom version, profile, port). No secrets, no credentials, no DB connection strings (dev profile uses H2 in-memory). Logs are local-only, not uploaded to CI. |
| T-78-11 | Spoofing (image identity) | Local image confused with release image | accept | Local tag `ctc-manager:phase78-local` is deliberately distinct from `release.yml`'s `ghcr.io/{owner}/ctc-manager:{version}` to prevent accidental local-image push. |

ASVS Level 1: No new auth surface, no PII flow, no data persistence beyond the ephemeral in-memory H2 of the dev profile inside the smoke container.
