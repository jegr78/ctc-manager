<bug_summary>
## Bug — diagnosed in `.planning/debug/59-tx-rollback-on-preview.md`

1. `DriverSheetImportController.preview()` is annotated `@Transactional(readOnly = true)` and the controller delegates to `DriverSheetImportService.preview(sheetUrl)`, which is also `@Transactional(readOnly = true)` (DriverSheetImportService.java:65). Spring's AOP proxy opens the outer transaction at the service boundary.
2. `preview()` iterates over year-numbered tabs and calls `buildTabPreview(spreadsheetId, "<year>")`. `buildTabPreview` is a private method — no `@Transactional` of its own, runs inside the outer transaction.
3. For a legacy `^\d{4}$` tab (e.g. tab name `"2025"`), `buildTabPreview` calls `seasonManagementService.findUnique(year)` (DriverSheetImportService.java:231).
4. `SeasonManagementService.findUnique(int year)` is currently `@Transactional(readOnly = true)` with default `Propagation.REQUIRED`. It joins the outer transaction. When `seasonRepository.findByYear(year)` returns 2+ hits (dev seed has `Test-Season 2025` year=2025 number=98 + the user's real 2025 season), it throws `BusinessRuleException(...)`.
5. **Spring's `TransactionInterceptor` for `findUnique` calls `doSetRollbackOnly()` on the SHARED outer transaction BEFORE the exception propagates back to Java.**
6. `buildTabPreview`'s `catch (BusinessRuleException ex)` (DriverSheetImportService.java:241-244) handles the exception correctly at the Java level, populating `ambiguousReason`. Execution continues. `preview()` returns a valid `DriverSheetImportPreview`.
7. At Spring's AOP commit boundary for `preview()`, the transaction is found `rollbackOnly=true`. `AbstractPlatformTransactionManager.processCommit()` throws `UnexpectedRollbackException`.
8. The controller does not catch `UnexpectedRollbackException`, so `GlobalExceptionHandler` surfaces it as HTTP 500.

## Fix — Option C (selected by debug session)

Remove the `@Transactional(readOnly = true)` annotation from BOTH overloads:

- `findUnique(int year, int number)` — currently SeasonManagementService.java:107
- `findUnique(int year)`            — currently SeasonManagementService.java:124

Without the annotation, Spring's `TransactionInterceptor` is no longer wrapping these method calls — when they throw `BusinessRuleException`, no `setRollbackOnly` side-effect occurs on whatever ambient transaction is active. The methods still see a valid Hibernate session (OSIV is enabled, plus the caller's `@Transactional` provides a session for repository calls). `seasonRepository.findByYearAndNumber` and `seasonRepository.findByYear` already declare their own implicit query transactions; removing the wrapper is safe.

**Rationale (mirrors Phase 58 prior art):**
> "Bridge uses findByType (Optional) instead of findRegularPhase to avoid transaction rollback-only poisoning."
The pattern is identical: eliminate the Spring AOP interception boundary that triggers `setRollbackOnly` on a "business signal" exception.

## Why the existing IT did not catch this

`DriverSheetImportServiceIT` is annotated `@Transactional` at the class level (line 50). The test framework wraps the entire method in one outer transaction that is auto-rolled-back at end-of-test. `findUnique`'s `BusinessRuleException` does set `rollbackOnly=true` on this test transaction, but **the test framework was already going to roll back** — Spring never attempts a real commit, so `UnexpectedRollbackException` is never thrown. The bug only surfaces in production where the controller's `preview()` actually commits its own transaction.

## What this plan adds for CI protection

A NEW IT — `DriverSheetImportServiceTransactionIT` — that intentionally omits class-level `@Transactional`. Each call to `driverSheetImportService.preview()` opens its own real transaction at the Spring AOP boundary and commits it. With the unfixed code: the test fails with `UnexpectedRollbackException`. With the fix: the test passes. Manual `@AfterEach` cleanup (`seasonRepository.deleteById`) compensates for the absent test-framework rollback.
</bug_summary>
