Private helper `writeJson(Map<String, Long> m)` uses `backupObjectMapper.writeValueAsString(m)`; on `JsonProcessingException` re-throws as `IllegalStateException` (the entire import is rolling back anyway in the failure case, and the success case cannot have unserializable Long maps). After `repository.save(audit)`, `log.info("Audit row written: id={}, success={}, totalRestored={}", saved.getId(), success, tableCountsRestored.values().stream().mapToLong(Long::longValue).sum())`. Return the saved entity. **No try-catch around the save itself** — REQUIRES_NEW is the entire mechanism; the caller MUST be ready to log+swallow a save failure to keep the outer rollback path running (this is documented in Plan 06's failure-handler design but the service stays simple).
