    ## 2. 24h retention semantics

    Documents:
      - Both `data/.import-backups/<ts>/uploads-old/` (Phase 75 carry-forward; previous upload tree) AND `data/.import-backups/<ts>/auto-backup-before-import.zip` (Phase 76 new) live under the same `<ts>` directory.
      - Both retained 24 hours by convention; operator cleans up older directories with the example command:
        ```
        find data/.import-backups -mindepth 1 -maxdepth 1 -type d -mtime +1 -exec rm -rf {} +
        ```
      - On Windows, the same cleanup uses `Get-ChildItem -Directory | Where-Object { $_.LastWriteTime -lt (Get-Date).AddDays(-1) } | Remove-Item -Recurse -Force` from PowerShell inside `data/.import-backups/`.

    ## 3. Audit-id query SQL

    Documents:
      - The `data_import_audit` table location and the example query:
        ```sql
        SELECT id, executed_at, executed_by, schema_version, success,
               source_filename,
               LEFT(table_counts_wiped, 100) AS wiped_preview,
               LEFT(table_counts_restored, 100) AS restored_preview
        FROM data_import_audit
        WHERE id = '<uuid-from-flash-message>';
        ```
      - How to interpret `success=false` + `table_counts_wiped='{}'` + `table_counts_restored='{}'`: this is the Phase 76 D-18 pre-import auto-backup failure signature — NO DB mutation occurred, but an audit trail of the attempt was preserved (REQUIRES_NEW propagation).
      - How to interpret `success=false` + non-empty counts: Phase 75 mid-restore-failure path — restore-loop rolled back; DB is back to pre-import state.
      - Column shapes: `table_counts_wiped` and `table_counts_restored` are LONGTEXT (Phase 72 D-09); JSON object `{ "seasons": 12, "drivers": 240, ... }`.

    ## 4. Concurrent-import behavior

    Documents:
      - Symptom: clicking `Execute Import` returns to `/admin/backup` with the flash error `"Another import is already running — please wait."` (verbatim per D-04) and HTTP status 409 (visible in browser DevTools Network tab).
      - Meaning: another admin tab/window or a separate operator is mid-import. The lock is held in-memory in the single JVM (D-03 — single-host deployment).
      - Resolution: refresh `/admin/backup` after ~30 seconds; once the yellow banner disappears, retry. If the lock appears to be stuck (banner visible for > 30 minutes with no log activity), restart the JVM — the in-memory lock dies with the process.

    ## 5. Read-only state during imports

    Documents:
      - Symptom: yellow banner reading `"Backup import in progress — write access is temporarily locked."` (verbatim per D-12) renders on every admin page; admin form submissions return HTTP 503 with the same English wording in the response body.
      - Meaning: an import is mid-execute; all writes under `/admin/**` except the whitelisted `/admin/backup/import-execute` are blocked.
      - Resolution: wait. The banner disappears automatically when the import finishes (success path) OR when the auto-backup-failure / wipe-failure paths roll back. The yellow banner is the trustworthy signal — when it's gone, writes work again.
