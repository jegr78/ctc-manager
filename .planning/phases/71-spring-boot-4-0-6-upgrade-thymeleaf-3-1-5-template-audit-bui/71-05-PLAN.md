<objective>
Install a lightweight Maven `exec-maven-plugin` build-guard that fails the build at the `validate`
phase if any future template re-introduces a `${...}` expression inside a `th:replace`,
`th:insert`, or `th:include` fragment-call argument under `src/main/resources/templates/`. The
regex matches `${pageTitle}` too — so the gate uses a two-stage pipeline: the broad regex finds
candidate lines, then `grep -v 'layout(${pageTitle}'` strips the canonical form. Anything that
survives is a regression and fails the build.

**Rationale:** Per CONTEXT.md D-05 the broad regex `th:(replace|insert|include)=".*\(.*\$\{.*\}.*\)"`
catches every `${...}` in a fragment-call argument. To preserve the broad invariant per D-05 and
avoid `grep -v` exclusions per D-03b (which is about TEMPLATE-side narrowing, NOT
build-script-side narrowing), the gate uses the negation idiom on the BUILD SIDE:

```bash
grep -rE 'th:(replace|insert|include)=".*\(.*\$\{.*\}.*\)"' src/main/resources/templates/ \
  | grep -v 'layout(${pageTitle}'
```

Post-Plans-01-02 the pipeline is empty by construction. A future contributor adding back any
other `${expr}` shape (ternary, Elvis, string-concat, plain `${otherVar}`) immediately fails the
build at `validate` phase.

This plan depends on Plan 03 (the platform must be on 4.0.6 first so the gate runs against the
final state) and is parallel to Plan 04 (different file: pom.xml vs test sources/resources).

Purpose: PLAT-07 — fulfill the forward-commitment from v1.9 Phase 67 D-06 to install a build-time
guard against the fragment-parameter-expression pattern.

<!-- exec-maven-plugin docs -->

`org.codehaus.mojo:exec-maven-plugin` is transitively available via the Spring Boot parent (per
CONTEXT.md D-17). The plugin's `exec:exec` goal accepts:
- `<executable>` — the command (e.g., `bash`)
- `<arguments>` — list of `<argument>` entries passed to the executable

The `exec:exec` goal is preferred over `exec:java` because we are invoking a shell, not a Java
main method.

<!-- Working-directory note -->

By default, `exec:exec` runs in `${project.basedir}` — the Maven project root. Relative paths
like `src/main/resources/templates/` resolve correctly from that working directory. No
`<workingDirectory>` override is needed.

**Why this regex + filter (per CONTEXT.md D-05, D-03b, and PATTERNS.md):**
- The broad regex catches ANY `${...}` in a `th:replace`/`th:insert`/`th:include` fragment-call
  argument. Post-Plans 01 + 02, the ONLY legitimate match is `layout(${pageTitle}, ~{::section})`
  in the 17 affected templates.
- The `grep -v 'layout(${pageTitle}'` filter strips those canonical lines. Anything that survives
  is by definition a regression.
- D-03b ensures `site/matchday.html` line 10 was refactored to `matchCardBody(race)` (no `${...}`),
  so the broad regex does NOT need a per-line carve-out.
- The CDATA wrapper protects the shell script (containing `>`, `<`, `&`, `$`) from XML parsing.
- `|| true` after the first grep prevents the pipeline from short-circuiting when grep finds zero
  matches (grep returns exit 1 on no-match by default).
- `exit 1` from the bash wrapper causes Maven to treat the execution as FAILED, halting the build
  during the `validate` phase — before compile, test, or any other phase runs.

**Why `<phase>validate</phase>` (per D-06):**
- `validate` is the FIRST phase of the Maven default lifecycle. The guard fails fast, giving the
  contributor sub-10s feedback before compile begins.

**Why NOT a Maven Enforcer custom rule (per D-07):**
- Custom Enforcer rules require their own Maven module. The `exec-maven-plugin` grep-gate ships
  the same outcome in ~30 lines of POM XML. The REQUIREMENTS § "Out of Scope" table explicitly
  rejects the Enforcer custom-rule approach.

**No new `<dependency>` introduced** (per D-17): `org.codehaus.mojo:exec-maven-plugin` is already
in the Spring Boot parent's pluginManagement section; declaring it under `<build><plugins>`
without an explicit `<version>` inherits the parent-managed version.

**Placement check:** the new `<plugin>` MUST be inside the `<build><plugins>` element of the
root pom, NOT inside the `<profiles><profile><build><plugins>` block (which is e2e-profile-only).
This way the guard runs unconditionally on every Maven invocation. Insert immediately after the
jacoco-maven-plugin closing `</plugin>` tag and before the closing `</plugins>` of root build.

**Encoding hardening:** avoid using literal `§` or `—` (em-dash) inside the CDATA section to
prevent any potential XML-parser or Maven-output-encoding fragility. The diagnostic uses only
7-bit-ASCII characters.
