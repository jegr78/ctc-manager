Install a lightweight Maven `exec-maven-plugin` build-guard that fails the build at the `validate`
phase if any future template re-introduces a `${...}` expression inside a `th:replace`,
`th:insert`, or `th:include` fragment-call argument under `src/main/resources/templates/`.

**Regex CORRECTED 2026-05-11 per plan-checker BLOCKER #1**: the original CONTEXT.md D-05 regex
(`th:(replace|insert|include)=".*\(.*\$\{.*\}.*\)"`) anchored on `)"` and matched ZERO real
offenders despite 17+ visible broken lines, because real Thymeleaf lines end with `)}"` (close-paren
+ outer-brace + quote, the `}` closing the `~{...}` fragment-expression wrapper). The corrected
regex `th:(replace|insert|include)="~\{[^"]*\(.*\$\{.*\}.*\)\}"` correctly matches all 18 offenders
(10 admin + 8 site) pre-Plans-01/02. After Plans 01 + 02 land, only the canonical
`layout(${pageTitle}, ~{::section})` form remains — the build-guard uses `grep -v` to strip those
canonical lines, leaving zero offenders post-phase.

**Rationale:** Per CONTEXT.md D-05 (revised) the broad regex catches every `${...}` in a
fragment-call argument shaped as the standard Thymeleaf `~{...(...)}` wrapper. To preserve the
broad invariant per D-05 and avoid `grep -v` exclusions per D-03b (which is about TEMPLATE-side
narrowing, NOT build-script-side narrowing), the gate uses the negation idiom on the BUILD SIDE:

```bash
grep -rE 'th:(replace|insert|include)="~\{[^"]*\(.*\$\{.*\}.*\)\}"' src/main/resources/templates/ \
  | grep -v 'layout(${pageTitle}'
```

Post-Plans-01-02 the pipeline is empty by construction (independently verified 2026-05-11: 0 of 18
remaining lines, all 18 were the canonical `layout(${pageTitle}` form). A future contributor adding
back any other `${expr}` shape (ternary, Elvis, string-concat, plain `${otherVar}`) immediately
fails the build at `validate` phase.

<!-- exec-maven-plugin docs -->

`org.codehaus.mojo:exec-maven-plugin` is transitively available via the Spring Boot parent (per
CONTEXT.md D-17). The plugin's `exec:exec` goal accepts:
- `<executable>` — the command (e.g., `bash`)
- `<arguments>` — list of `<argument>` entries passed to the executable

The `exec:exec` goal is preferred over `exec:java` because we are invoking a shell, not a Java
main method.

<!-- Working-directory note -->

By default, `exec:exec` runs in `${project.basedir}` — the Maven project root. Relative paths
like `src/main/resources/templates/` resolve correctly from that working directory. No
`<workingDirectory>` override is needed.

**Why this CORRECTED regex (per CONTEXT.md D-05 revised 2026-05-11, plan-checker BLOCKER #1):**

The regex `th:(replace|insert|include)="~\{[^"]*\(.*\$\{.*\}.*\)\}"` anchors on the real
Thymeleaf fragment-call shape:
- `th:(replace|insert|include)=` — the attribute name
- `"~\{` — opening of the fragment-expression wrap `"~{`
- `[^"]*` — anything up to the inner `(` that does NOT cross the outer `"` quote boundary
- `\(.*\$\{.*\}.*\)` — the call's parameter list containing at least one `${...}` expression
- `\}"` — close-brace of the `~{...}` wrap + closing `"` of the HTML attribute

The ORIGINAL regex (`th:(replace|insert|include)=".*\(.*\$\{.*\}.*\)"`) was broken because it
anchored on `)"` (close-paren + quote) — but real lines end with `)}"` (close-paren + outer-brace
+ quote, the `}` being the closer of the outer `~{...}` wrap). Independent verification 2026-05-11:
- Original regex against full templates/ tree: 0 matches (broken)
- Corrected regex against full templates/ tree (pre-Plans-01/02): 18 matches (real)

The `grep -v 'layout(${pageTitle}'` filter strips the canonical post-fix form. Anything that
survives is by definition a regression.

D-03b ensures `site/matchday.html` line 10 was inlined (matchCardBody fragment body inlined directly,
no fragment call), so the corrected regex does NOT need a per-line carve-out.

**Why the CDATA wrapper:** protects the shell script (containing `>`, `<`, `&`, `$`) from XML parsing.

**Why `|| true`:** prevents the pipeline from short-circuiting when grep finds zero matches
(grep returns exit 1 on no-match by default).

**Why `exit 1`:** causes Maven to treat the execution as FAILED, halting the build during the
`validate` phase — before compile, test, or any other phase runs.

**Why `<phase>validate</phase>` (per D-06):**
- `validate` is the FIRST phase of the Maven default lifecycle. The guard fails fast, giving the
  contributor sub-10s feedback before compile begins.

**Why NOT a Maven Enforcer custom rule (per D-07):**
- Custom Enforcer rules require their own Maven module. The `exec-maven-plugin` grep-gate ships
  the same outcome in ~30 lines of POM XML. The REQUIREMENTS § "Out of Scope" table explicitly
  rejects the Enforcer custom-rule approach.

**No new `<dependency>` introduced** (per D-17): `org.codehaus.mojo:exec-maven-plugin` is already
in the Spring Boot parent's pluginManagement section; declaring it under `<build><plugins>`
without an explicit `<version>` inherits the parent-managed version.

**Placement check:** the new `<plugin>` MUST be inside the `<build><plugins>` element of the
root pom, NOT inside the `<profiles><profile><build><plugins>` block (which is e2e-profile-only).
This way the guard runs unconditionally on every Maven invocation. Insert immediately after the
jacoco-maven-plugin closing `</plugin>` tag and before the closing `</plugins>` of root build.

**Encoding hardening:** avoid using literal `§` or `—` (em-dash) inside the CDATA section to
prevent any potential XML-parser or Maven-output-encoding fragility. The diagnostic uses only
7-bit-ASCII characters.

```bash

<html xmlns:th="http://www.thymeleaf.org"
      th:replace="~{admin/layout :: layout(${foo.bar}, ~{::section})}">
```

matches, the regex is broken — fail the smoke immediately with a diagnostic.
