**Gathered:** 2026-05-02

## Phase Boundary

Mirror the admin-side phase/group model on the public static site. The public site (generated by `SiteGeneratorService` into `docs/site/`) becomes phase- and group-aware analogous to `templates/admin/season-detail.html`.

**Lieferumfang per ROADMAP-Anforderungen:**

1. **Phase tabs on the public site** — every season's public landing renders one tab per existing phase (REGULAR / PLAYOFF / PLACEMENT). PLAYOFF tab links to existing `/season/{slug}/playoff.html` (no new bracket page). Single-REGULAR-LEAGUE seasons render identically to today (no tab chrome — SC4 hard backward-compat invariant).

2. **GROUPS-layout per-group standings + combined view** — when a phase is GROUPS-layout, sub-tabs surface per-group standings (one HTML file per group) plus a Combined-View default. Mirrors admin Phase 60 D-30/D-32 (Combined-View shows Group column; per-group view hides it; Buchholz only on per-group + Swiss).

3. **Phase- and group-aware templates** — `templates/site/standings.html` (today: zero phase/group references) becomes the canonical phase-aware template. Phase awareness extends to all four other public page types (matchdays, driver-ranking, team-profile, driver-profile) per the scope decisions below.

4. **`SiteGeneratorService` no longer LEAGUE-only** — today the backend is partially phase-aware (`seasonPhaseService.findRegularPhase(...)` at line 191) but template comments admit "Phase-aware standings via REGULAR phase (templates stay LEAGUE-shaped)". Phase 62 finishes the cutover at the template + URL layer.

5. **Regression test for ≥1 GROUPS season + ≥1 multi-phase season** in the sitegen test suite (SC5). Surefire integration test, not Playwright E2E.

**Tracked Behavior Changes (must call out at PR/Release time, mirror Phase 61 D-23 pattern):**

- **Alltime aggregation expanded from REGULAR-only to cross-phase per season (D-19)** — `alltime-standings.html` and `alltime-driver-ranking.html` numbers will change for any historical season that has a PLAYOFF / PLACEMENT phase. `SiteGeneratorService.generateAlltimeStandings/DriverRanking` and the underlying `StandingsService.calculateAlltimeStandings` / `DriverRankingService.calculateAlltimeRanking` paths must be updated accordingly. Today's alltime numbers are visible to the public — release notes must call out the recomputation.
- **New per-phase URLs introduced** — `standings-{phaseSlug}.html`, `standings-{phaseSlug}-group-{groupSlug}.html`, `matchdays-{phaseSlug}.html`, `driver-ranking-{phaseSlug}.html` (and group variants where applicable). Existing legacy URLs remain canonical defaults; new variants are additive.

**Übernommen aus früheren Phasen (nicht erneut diskutiert):**

- **Phase 58 D-04 / D-05** — Combined-View = flache `List<TeamStanding>` mit nullable `group`-Field; `StandingsService.calculateStandings(phaseId, groupId)` mit nullable groupId. Reusable as-is.

## Implementation Decisions

### URL / file scheme (D-01..D-04)

- **D-01: Phase-suffixed flat-file scheme.** Per-phase pages live at `/season/{slug}/standings-{phaseSlug}.html`. Per-group pages at `/season/{slug}/standings-{phaseSlug}-group-{groupSlug}.html`. Legacy `/season/{slug}/standings.html` stays as REGULAR-default (combined for GROUPS-layout REGULAR). Same suffix pattern for `matchdays-*.html` and `driver-ranking-*.html` (D-12). Flat directory = simpler cross-link relativization than the admin-style `/phase/{x}/group/{y}/` hierarchy. Mirrors today's `slugify(matchday.label) + ".html"` flatness.

- **D-02: `phaseSlug` = lowercased PhaseType name.** `regular` / `playoff` / `placement`. Stable across phase-label edits, predictable for cross-links, no slug collisions (UNIQUE phaseType per season already enforced by Phase 56 D-03 `UNIQUE (season_id, phase_type)`). Phase label remains visible in tab text + page heading; only the URL is type-based.

- **D-03: `groupSlug` = `slugify(group.name)`.** Matches existing site `slugify` convention (driver, team, matchday). Group rename → 404 on old bookmark — acceptable for non-canonical pages (groups are user-renameable but typically stable; the canonical landing is the per-phase combined view, not the per-group one).

- **D-04: Legacy `/season/{slug}/standings.html` renders REGULAR-combined with phase-tab row on top.** Existing URL stays canonical and bookmarkable. Tab row appears only when ≥2 phases or current phase is GROUPS-layout (D-06). Single-REGULAR-LEAGUE seasons render with no tab chrome → byte-for-byte indistinguishable from today (clean SC4 satisfaction). All cross-links from `teams.html`, `archive.html`, footer, sub-nav keep working unchanged.

### Phase / group tab UX (D-05..D-08)

- **D-05: Tabs styled native to public-site, not mirroring admin's pill-tab look.** Reuse the existing `.subnav` pattern from `site/layout.html` for the phase-tab row (and an analogous secondary row for group sub-tabs). Tighter integration with current dark theme; one less new CSS dialect. Tab anchor color uses existing `.entity-link` accent (#4fc3f7) for active state.

- **D-06: Tab visibility = hide when redundant.** Phase-tab row renders only when season has **≥2 phases**. Group-sub-tab row renders only when current phase is **GROUPS-layout**. Single-REGULAR-LEAGUE → no tab rows at all → page identical to today (SC4 invariant). Single-REGULAR-GROUPS (the 2023-test-data shape) → group sub-tab row visible, no phase row.

- **D-07: Tab anchors = inter-file links, not in-page anchors.** Each phase variant is its own static HTML file (per D-01). Tab nav at the top of each variant page links to sibling variants via relative paths. No JS toggle, no `:target` selectors — pure static HTML. Aligns with the rest of the static site's "one URL per state" philosophy.

- **D-08: PLAYOFF tab = direct link to existing `/season/{slug}/playoff.html`.** No new `standings-playoff.html` is generated for the bracket. Phase 60 D-41 (admin Playoff URL stable) extends here — bracket page already exists, no rewrite needed. PLAYOFF tab visually consistent with REGULAR tab in the row, but its href points to the playoff bracket page rather than a `standings-playoff.html`. SC3 requirement "PLAYOFF-phase tab on the public site reaches the playoff bracket / final standings" is satisfied by the link.

### Phase awareness scope (D-09..D-15)

- **D-09: Sub-nav stays coarse.** `site/layout.html` sub-nav links continue to point at legacy URLs (`standings.html` / `matchdays.html` / `driver-ranking.html` / `playoff.html`). `currentPage` stays as `'standings'` / `'matchdays'` / `'driver-ranking'` across all per-phase variants — active highlighting works the same. Phase-tab navigation lives WITHIN the page, not in sub-nav. Reduces shared-layout churn.

- **D-10: matchdays.html → tabs + suffixed files.** `/season/{slug}/matchdays.html` stays as REGULAR-default; `matchdays-{phaseSlug}.html` for per-phase variants; `matchdays-{phaseSlug}-group-{groupSlug}.html` for GROUPS phases. Consistent URL scheme with standings. Per-phase variant lists only matchdays of that phase (and group, if applicable).

- **D-11: driver-ranking.html → aggregated stays as legacy default.** Today's cross-phase aggregation (`driverRankingService.aggregateAcrossPhases(phaseIds, season.getId())`) stays at the legacy URL — SC4-clean (no behavior change on the canonical URL). Tab row on top: "All phases" (active default) | per-phase tabs. Per-phase variants: `driver-ranking-{phaseSlug}.html`. Cross-phase is the most-used view; per-phase is the drill-down.

- **D-12: Per-phase URL pattern is uniform across the three list-page types.** `standings-*.html` / `matchdays-*.html` / `driver-ranking-*.html` all follow the same `-{phaseSlug}[-group-{groupSlug}].html` suffix structure. One mental model for users; one helper-class pattern for the generator (see D-21).

- **D-13: team-profile.html → phase breakdown only when ≥2 phases.** Team participated in only REGULAR phase → page renders identically to today (SC4 backward-compat). Team participated in REGULAR + PLAYOFF (or +PLACEMENT) → new "Phase Breakdown" section appears (e.g. "Regular: 5th, 47pts | Playoff: SF exit"). The standings panel itself shows the **combined-view standing** (i.e. team's position in `calculateStandings(REGULAR, group=null)`) — D-14.

- **D-14: team-profile standings panel = combined-view standing.** When the team is in a GROUPS-layout REGULAR phase, the panel uses `calculateStandings(REGULAR-phase, groupId=null)` — the team's combined-view position. Per-group standing surfaces only via the breakdown section (when ≥2 phases). Matches the legacy default page; consistent UX. No combinatorial team-profile-{group}.html.

- **D-15: driver-profile.html → per-phase results section only when ≥2 phases.** Driver participated in only REGULAR phase → page renders identically to today. Otherwise: results sectioned by phase ("Regular Season Results" / "Playoff Results"). Useful when phases have different formats/scoring (e.g. PLAYOFF often non-points-based).

### Cross-link strategy (D-16..D-17)

- **D-16: All standings/matchdays/driver-ranking variants link to single team-profile.html / driver-profile.html.** No team-profile-regular.html / team-profile-playoff.html proliferation. Per-phase data lives in the breakdown section (D-13/D-15), not in URL forks. Avoids combinatorial explosion of files and cross-link surface.

- **D-17: Combined-View is the default landing inside a GROUPS-layout phase.** Mirroring admin Phase 60 D-30: `standings-regular.html` (or `standings.html` for legacy) of a GROUPS-REGULAR season = Combined view (all teams flat, with Group column showing group affiliation). Per-group sub-tabs link to `standings-regular-group-{groupSlug}.html` (no Group column; Buchholz column appears only when format=Swiss per D-33 carry-over). Visual + URL parity with admin.

### Alltime + behavior change (D-18..D-19)

- **D-18: Alltime pages still aggregate cross-season; data source changes from REGULAR-only to all-phases.** `alltime-standings.html` / `alltime-driver-ranking.html` stay structurally identical — same cross-season aggregation, same ranking algorithm. Only the per-season input data shifts from "REGULAR-phase points" to "sum of all phases' points per season".

- **D-19: TRACKED BEHAVIOR CHANGE — alltime numbers will recompute and may shift visibly.** Any historical season that has a PLAYOFF (or PLACEMENT) phase will now contribute its playoff points to alltime totals. Users with bookmarks / mental-models of today's alltime numbers will see different totals after Phase 62 ships. Must be called out in PR description and release notes (mirror Phase 61 D-23 pattern). Implementation: `StandingsService.calculateAlltimeStandings(productionSeasons)` (and the driver-ranking equivalent) must accept all phases per season, not just REGULAR. Planner verifies whether new service methods are needed or existing signatures suffice.

### SiteGeneratorService refactoring (D-20..D-21)

- **D-20: Extract per-page generators into helper classes.** Today's `SiteGeneratorService` has 7+ `generateX` private methods inlined in one class (~870 LOC). With per-phase variants multiplying that, the service would balloon. Extract `StandingsPageGenerator`, `MatchdaysPageGenerator`, `DriverRankingPageGenerator`, `TeamProfilePageGenerator`, `DriverProfilePageGenerator` as separate `@Service @RequiredArgsConstructor` Spring beans in `org.ctc.sitegen` package. `SiteGeneratorService` becomes orchestrator: per-season loop, calls each helper, aggregates `GenerationResult`. Cleaner boundary, easier per-page-type unit testing. Public-site is mature enough to absorb the abstraction; v1.6 already gestures toward separation via the `record` types defined inside `SiteGeneratorService`.

- **D-21: PLAYOFF-bracket page generation stays inside `SiteGeneratorService` (or in its own helper).** The `generatePlayoffBracket` method is largely unchanged by Phase 62 — it already resolves `playoff.getPhase()` (Phase 58/60 wiring). Planner decides whether to extract it as a `PlayoffBracketPageGenerator` helper for symmetry or leave it inline. Bracket layout itself is not in Phase 62 scope.

### Empty-state + edge cases (D-22..D-23)

- **D-22: Empty-state per phase = render with 0-point roster + banner.** Mirror Phase 60 D-36: when a phase has 0 race results yet, generate the standings page anyway with the full `PhaseTeam`-roster showing 0 points; banner reads "No results recorded yet." Predictable structure; users see who's competing even before the season starts. Consistent with the admin-side pattern. Applies to per-phase variants (`standings-playoff.html` for a freshly-created PLAYOFF phase). Tab in nav stays visible.

- **D-23: PLAYOFF-phase tab visibility before bracket exists.** PLAYOFF phase exists (record in `season_phases`), but no `Playoff` row yet (bracket not created) → PLAYOFF tab still renders, links to `playoff.html` which already handles empty-bracket gracefully. PLAYOFF phase exists with a `Playoff` row but no race results → bracket renders with TBD slots (today's behavior). No new "phase exists but bracket missing" empty-state needed.

### Test strategy (D-24..D-26)

- **D-24: SC5 regression test = Surefire integration test, not Playwright E2E.** Extend existing `SiteGeneratorIntegrationTest` pattern: `@SpringBootTest` with H2, run `siteGeneratorService.generate()`, assert generated HTML files exist (`Files.exists`) + assert key fragments via Jsoup parsing (e.g. presence of phase tab row, group sub-tab row, expected standings cell values). Fast, runs in normal `./mvnw verify` (Surefire), no Playwright dependency. Mirrors v1.6 sitegen test approach + Phase 61 D-09 V6MigrationTest "Surefire not Failsafe" choice for similar regression-gate tests.

- **D-25: Test fixtures = reuse Phase 59 D-09 GROUPS-2023 fixture from `TestDataService`.** Battle-tested in admin-side tests; single source of truth for GROUPS test data. Multi-phase coverage: extend an existing fixture with a PLAYOFF phase (or use the existing fixture if it already has one). Plus one fixture covering the legacy single-REGULAR-LEAGUE case (for SC4 backward-compat regression). Test-only `T-`-prefix isolation via `feedback_test_data_isolation` is implicit since reused fixtures already follow it.

- **D-26: Visual + accessibility verification scope = all 4 phase-aware page types + tab a11y.** `playwright-cli` Desktop + Mobile (CLAUDE.md mandate, `feedback_playwright_cli`) on standings, matchdays, driver-ranking, team-profile, driver-profile — for both LEAGUE single-phase (regression check) and GROUPS multi-phase (new behavior). Tab nav semantics: `role="tablist"` on the row, `role="tab"` + `aria-selected` on each anchor, `aria-controls` pointing at the page's main `<section>` id. Keyboard navigation: tabs are anchor links so native Tab-key focus + Enter activation works; no JS arrow-key handler needed. Skip-link continues to work as today.
