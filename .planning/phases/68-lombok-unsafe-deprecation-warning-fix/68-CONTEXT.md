# Phase 68: Lombok Unsafe Deprecation Warning Fix - Context

**Gathered:** 2026-05-07
**Status:** Ready for planning

<domain>
## Phase Boundary

Eliminate the JDK 24+/25 build-time warnings emitted by `lombok.permit.Permit` calling the terminally-deprecated `sun.misc.Unsafe::objectFieldOffset`:

```
WARNING: A terminally deprecated method in sun.misc.Unsafe has been called
WARNING: sun.misc.Unsafe::objectFieldOffset has been called by lombok.permit.Permit
WARNING: Please consider reporting this to the maintainers of class lombok.permit.Permit
WARNING: sun.misc.Unsafe::objectFieldOffset will be removed in a future release
```

Both `./mvnw verify` and `./mvnw spring-boot:run -Dspring-boot.run.profiles=dev` must complete WITHOUT these 4 warning lines.

**Current baseline (verified via codebase scout):**
- Java 25 (Eclipse Temurin) per `pom.xml:<java.version>25</java.version>`.
- Spring Boot 4.0.5 starter parent.
- Lombok 1.18.44 (resolved transitively from the Spring Boot parent — `~/.m2/repository/org/projectlombok/lombok/1.18.44/` confirms).
- Project pom.xml already references `${lombok.version}` in the annotation-processor path declaration — so a property override is the canonical hook.

**In scope:**
- Pin a Lombok version that has switched `Permit` away from `sun.misc.Unsafe` (toward `MethodHandles.privateLookupIn` or equivalent).
- Override via `<lombok.version>` property in pom.xml `<properties>` block.
- Confirm warnings are gone from `./mvnw verify` AND `./mvnw spring-boot:run` startup output.
- Confirm no behavior regression: `./mvnw verify` exits 0; tests-run count unchanged from Phase-67 baseline (1,231); JaCoCo BUNDLE LINE ≥ 0.82.

**Out of scope (deferred — see `<deferred>`):**
- Switching to a different annotation processor (Eclipse j2cl, etc.).
- Replacing Lombok with hand-written getters/setters or Java records.
- Spring Boot framework upgrade (CLAUDE.md "no breaking framework upgrade" implicit constraint).
- Java version downgrade (back to 21 LTS) — Java 25 is locked per CLAUDE.md.
- `--enable-native-access` / `--add-opens` JVM flag workarounds — band-aid, not fix.
- All other JVM deprecation warnings unrelated to Lombok.

</domain>

<decisions>
## Implementation Decisions

### Strategy (REVISED 2026-05-07 after RESEARCH.md HIGH-confidence finding)

> **Research outcome:** Lombok 1.18.46 (latest stable, master, and edge channel) all still call `sun.misc.Unsafe.objectFieldOffset` from `lombok.permit.Permit`. Upstream issue #3959 is open without maintainer response. **Pure version-pin cannot achieve the goal.** User decision (2026-05-07) selects **combined Path 1+2**: hygiene-bump AND JVM flag.

- **D-01:** **Lombok version pin to 1.18.46** via `<lombok.version>` property override in `pom.xml` `<properties>`. Hygiene value (other bugfixes since 1.18.44); does NOT silence the Permit/Unsafe warnings on its own (upstream-blocked).
- **D-02 (REVISED):** **JEP 498 JVM flag adopted in Surefire/Failsafe argLine.** Add `--sun-misc-unsafe-memory-access=allow` to the existing `<argLine>` in both `maven-surefire-plugin` and `maven-failsafe-plugin` configurations (already injects Mockito agent + `@{argLine}`). This is the JDK-25 OFFICIAL escape hatch introduced by JEP 498 — distinct from the older `--add-opens` / `--enable-native-access` band-aids that the original D-02 forbade. Until upstream Lombok #3959 lands, this is the cleanest way to mute the warning quartet. Document the flag with a `<!-- JEP 498 escape: silence Lombok #3959 Permit/Unsafe warnings until upstream fix lands -->` comment in pom.xml so the flag is removed once Lombok ships the Permit refactor.
- **D-03:** **No annotation-processor switch.** Keep `org.projectlombok:lombok`. Replacing Lombok would touch every `@Getter` / `@Setter` / `@RequiredArgsConstructor` / `@Slf4j` / `@Data` / `@NoArgsConstructor` use site — massive refactor that violates CLAUDE.md "Don't add features, refactor, or introduce abstractions beyond what the task requires."
- **D-04:** **Spring Boot stays at 4.0.5.** Independent Lombok pin via property override is the project's existing pattern (cf. `<playwright.version>`).

### Version selection (RESOLVED)

- **D-05 (RESOLVED):** **Lombok 1.18.46** (latest stable, released 2026-04-22). Confirmed via RESEARCH.md against upstream changelog + Maven Central. Hygiene-only — does NOT carry the Permit/Unsafe fix (none of stable / edge / master does as of 2026-05-07; tracked in upstream issue #3959). The bump still ships unrelated bugfixes accumulated since 1.18.44.
- **D-06:** **Compile-test gate.** The version must produce identical generated bytecode for the project's Lombok use sites — `./mvnw verify` exits 0 with `Tests run: 1231, Failures: 0` (Phase-67 baseline). If regression: revert and document.

  3. `./mvnw test -Dtest=DriverSheetImportServiceTest` — single test invocation as a Surefire-only smoke check.
- **D-08:** **Quantitative gate per context:** `<context-output> | grep -cE "sun\.misc\.Unsafe|lombok\.permit\.Permit"` returns **0**.

### Test impact

If bytecode differs in a way that breaks a test, that's a Lombok regression — log it and pin the previous version with explanatory comment.
- **D-10:** **JaCoCo gate** — coverage cannot regress. Property pin doesn't affect production code, so coverage stays at the Phase-67 baseline (~0.8561). JaCoCo gate (`./mvnw verify`) is the safety net.

<specifics>
## Specific Ideas

- **The 4 warning lines are emitted on stderr by the JDK runtime, not by Lombok itself.** The deprecation framework prints a `WARNING:` line whenever a terminally-deprecated method is called. So the fix path is upstream: change the *caller* (Lombok's `Permit` class). We can't suppress these via `-Xlint` — they're JDK runtime warnings.
- **The user's verbatim warning capture:**
  ```
  WARNING: A terminally deprecated method in sun.misc.Unsafe has been called
  WARNING: sun.misc.Unsafe::objectFieldOffset has been called by lombok.permit.Permit
  WARNING: Please consider reporting this to the maintainers of class lombok.permit.Permit
  WARNING: Please consider reporting this to the maintainers of class lombok.permit.Permit (this happens because reflection has been removed)
  WARNING: sun.misc.Unsafe::objectFieldOffset will be removed in a future release
  ```
  Note the exact strings — D-08 grep gate must use them.
- **Auto/Chain pipeline:** ran under `/gsd-discuss-phase 68 --auto --chain` until research (HIGH confidence) surfaced an upstream blocker — pure version-pin cannot achieve the goal because Lombok 1.18.46 / master / edge all still call `sun.misc.Unsafe.objectFieldOffset`. User chose Path 1+2 combined (hygiene-bump + JEP 498 flag) on 2026-05-07. CONTEXT.md was revised in place; planner now writes one plan with 4 tasks against the revised D-01..D-14.
- **Removal trigger for the JEP 498 flag (D-02-revised):** when upstream Lombok #3959 ships a stable release with the Permit refactor, the flag's TODO comment and the flag itself can be removed in a follow-up phase. The comment makes this discoverable.
- **Recommended-default selections** per `--auto` mode: D-01 property override (vs. JVM workaround or processor swap), D-04 keep Spring Boot stable, D-11 single plan with 3 tasks, D-13/D-14 strict (no half-fix shipping).

</specifics>
