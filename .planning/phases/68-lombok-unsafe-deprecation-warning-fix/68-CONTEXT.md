# Phase 68: Lombok Unsafe Deprecation Warning Fix - Context

**Gathered:** 2026-05-07
**Status:** Ready for planning

<domain>
## Phase Boundary

Eliminate the JDK 24+/25 build-time warnings emitted by `lombok.permit.Permit` calling the terminally-deprecated `sun.misc.Unsafe::objectFieldOffset`:

```
WARNING: A terminally deprecated method in sun.misc.Unsafe has been called
WARNING: sun.misc.Unsafe::objectFieldOffset has been called by lombok.permit.Permit
WARNING: Please consider reporting this to the maintainers of class lombok.permit.Permit
WARNING: sun.misc.Unsafe::objectFieldOffset will be removed in a future release
```

Both `./mvnw verify` and `./mvnw spring-boot:run -Dspring-boot.run.profiles=dev` must complete WITHOUT these 4 warning lines.

**Current baseline (verified via codebase scout):**
- Java 25 (Eclipse Temurin) per `pom.xml:<java.version>25</java.version>`.
- Spring Boot 4.0.5 starter parent.
- Lombok 1.18.44 (resolved transitively from the Spring Boot parent — `~/.m2/repository/org/projectlombok/lombok/1.18.44/` confirms).
- Project pom.xml already references `${lombok.version}` in the annotation-processor path declaration — so a property override is the canonical hook.

**In scope:**
- Pin a Lombok version that has switched `Permit` away from `sun.misc.Unsafe` (toward `MethodHandles.privateLookupIn` or equivalent).
- Override via `<lombok.version>` property in pom.xml `<properties>` block.
- Confirm warnings are gone from `./mvnw verify` AND `./mvnw spring-boot:run` startup output.
- Confirm no behavior regression: `./mvnw verify` exits 0; tests-run count unchanged from Phase-67 baseline (1,231); JaCoCo BUNDLE LINE ≥ 0.82.

**Out of scope (deferred — see `<deferred>`):**
- Switching to a different annotation processor (Eclipse j2cl, etc.).
- Replacing Lombok with hand-written getters/setters or Java records.
- Spring Boot framework upgrade (CLAUDE.md "no breaking framework upgrade" implicit constraint).
- Java version downgrade (back to 21 LTS) — Java 25 is locked per CLAUDE.md.
- `--enable-native-access` / `--add-opens` JVM flag workarounds — band-aid, not fix.
- All other JVM deprecation warnings unrelated to Lombok.

</domain>

<decisions>
## Implementation Decisions

### Strategy

- **D-01:** **Lombok version pin via `<lombok.version>` property override** in `pom.xml` `<properties>`. The pom.xml already uses `${lombok.version}` for the annotation-processor path declaration, so a property override is the project's existing extension hook — no plugin changes, no dependency-management overrides needed.
- **D-02:** **No JVM workarounds.** Do NOT add `--add-opens java.base/sun.misc=ALL-UNNAMED` or `--enable-native-access` flags to Surefire/Failsafe. Those are band-aids that suppress the symptom while leaving the root cause; the root cause is the Lombok version, fixable by upgrade.
- **D-03:** **No annotation-processor switch.** Keep `org.projectlombok:lombok` as the annotation processor. Replacing Lombok would touch every `@Getter` / `@Setter` / `@RequiredArgsConstructor` / `@Slf4j` / `@Data` / `@NoArgsConstructor` use site in the codebase — a massive refactor that violates CLAUDE.md "Don't add features, refactor, or introduce abstractions beyond what the task requires."
- **D-04:** **Spring Boot stays at 4.0.5.** Bumping the framework would introduce broader compatibility risk for a single warning-class fix. Pinning Lombok independently of the framework's BOM is the project's existing pattern (other deps follow the same property-override approach).

### Version selection

- **D-05:** **Version selection deferred to research.** RESEARCH.md must:
  1. Verify the latest stable Lombok release as of 2026-05-07.
  2. Identify the first version where `lombok.permit.Permit` switched from `sun.misc.Unsafe::objectFieldOffset` to `MethodHandles.privateLookupIn` (or equivalent JDK-current API).
  3. If no stable release has the fix yet, evaluate the "edge" / snapshot release path (Lombok ships edge artifacts via `https://projectlombok.org/edge-releases/`).
  4. Confirm Java 25 + Spring Boot 4.0.5 compatibility for the chosen version.
- **D-06:** **Compile-test gate.** The chosen version must produce identical generated bytecode for the project's Lombok use sites — `./mvnw verify` exits 0 with `Tests run: 1231, Failures: 0` (Phase-67 baseline). If regression: revert the property and document blocking issue.

### Verification surface

- **D-07:** **Three runtime contexts to verify warnings-clean:**
  1. `./mvnw verify` — full Maven build (compile + test). Capture stderr+stdout, grep for the 4 warning lines.
  2. `./mvnw spring-boot:run -Dspring-boot.run.profiles=dev` — application startup until "Started [App] in N seconds" line. Capture early stderr, grep for the 4 warning lines, then Ctrl-C.
  3. `./mvnw test -Dtest=DriverSheetImportServiceTest` — single test invocation as a Surefire-only smoke check.
- **D-08:** **Quantitative gate per context:** `<context-output> | grep -cE "sun\.misc\.Unsafe|lombok\.permit\.Permit"` returns **0**.

### Test impact

- **D-09:** **No production-source change.** Comments and `pom.xml` `<properties>` only. Generated bytecode SHOULD be identical (same Lombok feature surface, just internal Permit refactor). If bytecode differs in a way that breaks a test, that's a Lombok regression — log it and pin the previous version with explanatory comment.
- **D-10:** **JaCoCo gate** — coverage cannot regress. Property pin doesn't affect production code, so coverage stays at the Phase-67 baseline (~0.8561). JaCoCo gate (`./mvnw verify`) is the safety net.

### Plan structure

- **D-11:** **Single plan `68-01-PLAN.md`** with 3 tasks:
  1. Pin `<lombok.version>` in `pom.xml` `<properties>` to the version selected by RESEARCH.md.
  2. Run `./mvnw verify` and capture output; grep for the 4 warning lines (must be 0).
  3. Run `./mvnw spring-boot:run` startup and grep early stderr (must be 0); confirm app starts cleanly; Ctrl-C.
- **D-12:** **Atomic commit:** `fix(68): pin Lombok to <version> to remove sun.misc.Unsafe deprecation warning`. Single commit covers the property override.

### Rollback plan

- **D-13:** **If verify breaks:** revert the property line; document the failing version + the failure shape in `68-01-SUMMARY.md`; surface to the user as a blocker (don't ship a half-fixed phase).
- **D-14:** **If only ONE of the three runtime contexts (D-07) still emits warnings:** that's a partial success — accept and document if the project's typical workflow doesn't hit the affected context. Default: don't accept partial; the goal is "all four warning lines gone everywhere."

### Branch

- **D-15:** Stay on `gsd/v1.9-season-phases-groups` per CLAUDE.md branch protection.

### Claude's Discretion

- **D-16:** **Choice between latest stable vs. edge release** — left to RESEARCH.md based on the actual release status. Default toward stable; only escalate to edge if no stable release has the fix.
- **D-17:** **Property vs. dependencyManagement override style** — both work; planner picks the canonical Spring Boot project pattern (typically `<properties>` works because the parent BOM uses `${lombok.version}` indirection).

</decisions>

The deprecation warning is a JDK lint-class noise, not a build failure. Lombok's Permit refactor is a hardening for the JDK lifecycle — Lombok 1.18.40+ tracked the JDK 24 preview deprecation and started moving away from `sun.misc.Unsafe`; the question is which exact stable version landed the migration.

<specifics>
## Specific Ideas

- **The 4 warning lines are emitted on stderr by the JDK runtime, not by Lombok itself.** The deprecation framework prints a `WARNING:` line whenever a terminally-deprecated method is called. So the fix path is upstream: change the *caller* (Lombok's `Permit` class). We can't suppress these via `-Xlint` — they're JDK runtime warnings.
- **The user's verbatim warning capture:**
  ```
  WARNING: A terminally deprecated method in sun.misc.Unsafe has been called
  WARNING: sun.misc.Unsafe::objectFieldOffset has been called by lombok.permit.Permit
  WARNING: Please consider reporting this to the maintainers of class lombok.permit.Permit
  WARNING: Please consider reporting this to the maintainers of class lombok.permit.Permit (this happens because reflection has been removed)
  WARNING: sun.misc.Unsafe::objectFieldOffset will be removed in a future release
  ```
  Note the exact strings — D-08 grep gate must use them.
- **Auto/Chain pipeline:** Running under `/gsd-discuss-phase 68 --auto --chain` → planner writes one plan → executor pins property → verifier confirms warnings gone. Self-contained.
- **Recommended-default selections** per `--auto` mode: D-01 property override (vs. JVM workaround or processor swap), D-04 keep Spring Boot stable, D-11 single plan with 3 tasks, D-13/D-14 strict (no half-fix shipping).

</specifics>

<deferred>
## Deferred Ideas

- **Lombok elimination** (replace with Java records / hand-written getters / setters): mentioned as out-of-scope in `<domain>`. A separate quality phase if the team ever wants to drop the annotation-processor dependency entirely.
- **JVM `--add-opens` / `--enable-native-access` band-aids:** explicitly rejected per D-02. If Lombok upstream takes too long to fix, a workaround could be revisited — but the current default is "wait for the upstream fix and pin it."
- **Spring Boot framework upgrade** (4.0.6 → 4.1.x): tracked in Dependabot PR `dependabot/maven/org.springframework.boot-spring-boot-starter-parent-4.0.6` — out of scope here. Phase 68 is intentionally narrow.
- **Other JDK 25 deprecation warnings:** if any other warnings surface during this phase's verify runs (e.g., other libs calling deprecated APIs), file as separate phases. Phase 68 is laser-focused on the Lombok/Permit/Unsafe quartet.
- **Lombok release-channel decision lock-in** (stable vs. edge): tracked per-phase, not project-wide policy. A future phase could establish a project-wide rule if more lib-version edge cases come up.
