# Plan 114-03 — Summary

**Wave:** 2 | **Status:** complete | **Branch:** gsd/v1.17-guest-drivers

## What was built

Extended `DriverProfilePageGenerator` so a **pure guest** (a driver who appears only in a `RaceLineup` for a season, with no `SeasonDriver` row) gets a public driver-profile page (D-05). SCORE-02 page-existence hook.

### Implementation (`DriverProfilePageGenerator`)
- Extracted the entire per-driver render (results, phase breakdown, Context population, page write) into a private `writeDriverProfile(ctx, season, driver, team, seasonHasMultiplePhases, result)` so the first pass and second pass produce a **provably identical** page — only the team source differs.
- First pass unchanged behaviorally: iterates `seasonDriverRepository.findBySeasonId`, dedups via `generatedDriverIds`, passes `sd.getTeam()`.
- **New second pass:** `raceLineupRepository.findByRaceMatchdaySeasonId(season.getId()).stream().map(RaceLineup::getDriver).distinct()` enumerates every driver who appeared in the season. Reuses the SAME `generatedDriverIds` set (no second HashSet) — dual-role drivers already generated by the first pass are skipped (pitfall #4). For remaining lineup-only drivers, team is resolved via `findByDriverIdAndRaceMatchdaySeasonId(...).findFirst().map(rl -> rl.getTeam().getParentOrSelf())`; `team == null` is guarded by skipping (pitfall #6).
- No guest marker / badge / `*` / CSS / template change — Phase 115 boundary respected.

### Test (`DriverProfilePageGeneratorTest`)
- New `givenPureGuestDriver_whenGenerate_thenProfilePageExists` asserts the lineup-only guest `Test_Guest_1` gets a profile page under the Test-Season 2026 driver directory and the page body contains the PSN id.

## Deviation (important)
The plan's Task-1 test design assumed `siteGeneratorService.generate()` would emit the Test-Season guest page. It does **not**: `SiteGeneratorService.generate()` (line 81) filters out every season whose name contains "Test" from the public site. A Test-season guest can never appear via the full generator.

**Resolution:** the test drives the `DriverProfilePageGenerator` bean **directly** — builds a `GenerationContext` for Test-Season 2026 (resolved via `seasonRepository.findByYearAndNumber(2026, 99)`), instantiates a `GenerationResult`, and calls `generate(ctx, result)`. The test method is `@Transactional(readOnly = true)` to provide the Hibernate session that `SiteGeneratorService.generate()` (also `@Transactional(readOnly = true)`) normally supplies — without it the generator throws `LazyInitializationException` on `Matchday`/`Season`/`Phase` lazy associations. The slug is derived via `siteSlugger.slugify(season.getDisplayLabel())` (not hardcoded). This proves the D-05 second-pass logic exactly, against the real seeded guest, without weakening the production test-season filter.
