## PLANNER-DISCRETION DECISIONS

1. **`MatchdayPairingsForm` als separate DTO + sibling GET/POST-Endpoint
   `/admin/matchdays/{id}/edit-pairings` (NICHT in existing `MatchdayForm`
   integriert).** Rationale: `MatchdayForm` ist auf 3 Felder (label,
   sortIndex, seasonId) optimiert + wird vom existing `/admin/matchdays/save`
   Pfad gebunden. `pickDeadline` (LocalDateTime) + `scheduledWeekend`
   (String) sind Discord-Post-specific. Separate Form vermeidet Type-Spread
   und hält den existing Save-Flow unverändert. Inline-edit auf der
   Matchday-Detail-Seite wäre 3rd-Option (Planner-discretion in CONTEXT)
   aber: Inline-Edit ohne dedizierter Edit-Page erzeugt Race-Conditions
   mit dem GET-Refresh + ist nicht testbar via Playwright ohne JS-Trigger.

2. **`matchdayAnnouncementActive` als neue Boolean im Model neben dem
   existing `matchdayDiscordActive`.** Channel-Differentiation Discipline
   (operator-direktive 2026-05-25 [[feedback-discord-channel-types]]): die
   neue Pairings-Section gated nur auf `announcementWebhookUrl != null`,
   nicht auf den Race-Results-Forum-Webhook der existing Results/Power-
   Rankings-Buttons gated.

3. **`DiscordPostServiceMatchdayPairingsPreFlightTest`** als Mockito-Unit
   (kein Spring-Context, kein DB). Stub-Mode-Pattern aus Phase 95
   (`canPostMatchPreview*Test`).

4. **Default-Template** ist eine `private static final String
   DEFAULT_MATCHDAY_PAIRINGS_TEMPLATE` Konstante in `DiscordPostService`
   (analog `private static final String VS_EMOJI_FALLBACK` Pattern).
