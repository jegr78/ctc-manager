**Change 2: Replace containsSpringElTypeAccess() with context-aware detection in TemplatePreviewService.java**

Replace the entire `containsSpringElTypeAccess` method (lines 349-362) with this context-aware implementation that only checks for T() patterns inside `${...}` expression blocks:

```java
private boolean containsSpringElTypeAccess(String content) {
    int idx = 0;
    while ((idx = content.indexOf("${", idx)) != -1) {
        int end = content.indexOf('}', idx + 2);
        if (end == -1) break;
        String expr = content.substring(idx + 2, end);
        int tIdx = 0;
        while ((tIdx = expr.indexOf('T', tIdx)) != -1) {
            int next = tIdx + 1;
            while (next < expr.length() && expr.charAt(next) == ' ') {
                next++;
            }
            if (next < expr.length() && expr.charAt(next) == '(') {
                return true;
            }
            tIdx++;
        }
        idx = end + 1;
    }
    return false;
}
```

Key difference from original: Scans for `T(` pattern ONLY within `${...}` expression contexts, not the entire template string. This is the same approach already used by `containsOgnlStaticAccess()` (lines 364-378). CSS functions like `translateY()` and normal text like `T (Alpha)` are outside `${...}` blocks and are not evaluated by Thymeleaf, so they pose no security risk.

Security coverage maintained: SpEL injection via `T()` only works inside Thymeleaf expression contexts (`${...}`). Existing test `givenSpringElTypeAccess_whenValidate_thenRejectsTemplate` uses `${T(java.lang.Runtime)}` which IS inside `${...}` and will still be caught.
