## Copywriting Contract

Phase 30 introduces exactly one new user-visible string: the save-path security rejection message. All other copywriting is unchanged.

| Element | Copy |
|---------|------|
| Primary CTA | "Save Template" — existing, unchanged |
| Security rejection flash error | "Template contains unsafe expressions" |
| Preview security rejection | "Template contains unsafe expressions" (existing — already in preview path, now consistent with save path) |
| CSRF rejection | No user-visible copy — Spring Security returns 403; `csrfFetch()` prevents this by design (D-10) |
| Empty state heading | not applicable — no new list/table views |
| Empty state body | not applicable |
| Destructive confirmation | not applicable — no destructive actions in this phase |

Copy rationale for "Template contains unsafe expressions":
- Generic by design (D-08): no details about which pattern was blocked
- Consistent: same string on both save and preview paths
- Actionable implication: user must remove the flagged expression from the template
- Server-side detail: logged at `warn` level via `log.warn()` (D-09)

Source: CONTEXT.md D-08, D-09; RESEARCH.md `TemplateEditorController.save()` code example.

## Interaction Contract

### CSRF Token Delivery (SECU-03)

- CSRF meta tags (`<meta name="_csrf">`, `<meta name="_csrf_header">`) rendered conditionally in `layout.html` `<head>` using `th:if="${_csrf != null}"` guard
- `csrfFetch()` inline `<script>` placed directly below the meta tags in `<head>`
- `csrfFetch()` guards on both element presence AND non-empty content before injecting the header
- On prod/docker: meta tags present with token values → CSRF header injected on every AJAX POST
- On dev/local: meta tags absent (CSRF disabled) → fetch proceeds without CSRF header, no error
- No visible state change to the user — transparent to all UI

### Template Save Rejection (SECU-04)

- When save endpoint rejects an unsafe template: redirect back to template editor with `errorMessage` flash attribute
- Flash error rendered by existing `layout.html` `.alert.alert-error` block
- User remains on the template editor page (redirect preserves `?tab={templateType}` parameter)
- No modal, no confirmation dialog — single flash message is sufficient
- Error clears on next page load (standard flash attribute behavior)

### No New Components

This phase introduces no new HTML components, no new CSS classes, and no new JavaScript modules beyond the inline `csrfFetch()` function definition.
