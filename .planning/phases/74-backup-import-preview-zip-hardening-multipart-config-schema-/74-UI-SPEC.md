# Phase 74 — UI Design Contract

> Visual and interaction contract for the Backup Import (preview + confirm + upload) phase. Generated by gsd-ui-researcher, validated by gsd-ui-checker. Tokens are extracted from the project-owned `src/main/resources/static/admin/css/admin.css` — that file is the authoritative design system for this Thymeleaf admin SPA. ### 60/30/10 split for Phase 74

- **60 % dominant** — page body, paragraph copy, sidebar nav text, table cell text. Pure black (`#0a0a0a`) with off-white (`#d0d0d0`) text.
- **30 % secondary** — the 24 preview `.card` surfaces, the schema-match alert, the upload-form `.card`, the confirm-page warning callout. Charcoal `#141414` / `#1a1a1a` / `#3b2e0e`.
- **10 % accent** — exclusively the **sidebar active border** (`Backup` link on the left rail) and **focus rings** on file input + submit buttons. Sky blue `#4fc3f7`.

**Accent reserved for:**
1. The left-border indicator (`border-left: 3px solid var(--accent)`) on the active sidebar entry (`Backup` while on `/admin/backup`, `/admin/backup/import-preview`, `/admin/backup/import-confirm`).
2. Keyboard focus rings on `<input type="file">`, `<input type="checkbox">`, `<button>` (via `.btn:focus-visible`, `outline: 2px solid var(--border-focus)`).
3. Nothing else. No accent-coloured headings, no accent-coloured icons, no accent-coloured links inline.

### Destructive colour

`#d32f2f` (`.btn-danger`) is reserved for **one single button** in Phase 74: the `Execute Import` submit button on `admin/backup-confirm.html`. The `Import Backup` button on the landing page is NOT destructive — it merely opens the preview screen; it uses `.btn-primary` (blue).

### Delta-pill colour contract

| State | Class chain | Visual |
|-------|-------------|--------|
| `importedRows < currentRows` (data loss) | `badge alert-error` | Red text `#ef5350` on red-tinted background `#3a1b1b` |
| `importedRows >= currentRows` AND not both zero | `badge alert-success` | Green text `#66bb6a` on green-tinted background `#1b3a1b` |
| `currentRows == 0 && importedRows == 0` | `badge badge-inactive` | Grey text `#999` on `#222` background |

**Rationale:** red is the most visually consequential signal on the entire page (per CONTEXT D-03). The admin must spot data-loss-risk cards at a glance while scrolling the 24-card grid. Green and grey both recede; red claims attention.

**Pill content:** signed integer delta (`importedRows - currentRows`). Prefix `+` for gains is OPTIONAL but recommended (Thymeleaf: `${(card.importedRows - card.currentRows) > 0 ? '+' : ''} + ${card.importedRows - card.currentRows}`); negative values get a literal `-` from integer arithmetic.

**Contrast check:** all three combinations clear WCAG AA 4.5:1 on 12px+ text against the dark surface — verified empirically by the existing `.alert-success` / `.alert-error` / `.badge-inactive` already used on `admin/seasons.html`, `admin/import-preview.html`, and `admin/teams.html` for 6+ months.

### Empty-state copy

Phase 74 has no true empty states (the upload form is always present; the preview is never shown if the upload fails). However:

| Surface | Empty / Zero state | Copy |
|---------|---------------------|------|
| Preview card when `currentRows == 0 && importedRows == 0` | Card renders with grey neutral delta pill `0` | No special copy — the `0 → 0` line plus the neutral pill is self-explanatory |
| Header — `uploadFileCount == 0` | Renders `Uploads: 0 files` | Permissible (an export with no uploaded files is a valid edge case; not an error) |

### Error-state copy strategy

Phase 74 uses **redirect-with-Flash** for all server-side rejections, NOT inline error rendering. The chain is:

1. POST to `/admin/backup/import-preview` (multipart upload).
2. Validation fails (multipart-too-large / schema-mismatch / ZIP-safety).
3. Controller calls `redirectAttributes.addFlashAttribute("errorMessage", "<locked D-02 string>")`.
4. Redirect to `/admin/backup` (302).
5. Landing page renders the Flash via `admin/layout.html:83` (`<div th:if="${errorMessage}" class="alert alert-error" th:text="${errorMessage}">`).

**Inline form error** is used ONLY on `admin/backup-confirm.html` for the `acknowledged` checkbox (CONTEXT D-10): if `@AssertTrue` validation fails, the confirm page re-renders WITH `BindingResult` errors injected next to the checkbox via `.field-error` (the existing `admin.css:318` styling). No redirect, no Flash.

### Destructive actions

| Action | Surface | Confirmation contract |
|--------|---------|------------------------|
| `Execute Import` | `admin/backup-confirm.html` | Three-layer defence: (1) mandatory `@NotNull @AssertTrue Boolean acknowledged` server-side check (CONTEXT D-10); (2) `<form onsubmit="return confirm('Replace all operational data? This cannot be undone.');">` JS dialog (UX-extra); (3) destructive-coloured `.btn-danger` styling — red `#d32f2f`. In Phase 74 the actual mutation is stubbed — see "Stub-success Flash" string above. Phase 75 removes the stub Flash and runs the real wipe + restore. |

**Layout decisions:**

The controller does the delete and redirects. (The Research-doc skeleton showed a nested form-POST; this UI-SPEC overrides that — a `<a class="btn btn-secondary">` is simpler, accessible by keyboard, and avoids nesting `<form>` tags which is invalid HTML.)

#### `admin/backup-confirm.html` (NEW — Phase 74)

```
<h1>Backup Import — Confirm</h1>

<div class="alert alert-warning mb-md">     ← Destructive warning (yellow callout)
  This will delete ALL operational data and replace it with the contents
  of the uploaded backup. This action cannot be undone.

<form th:action="@{/admin/backup/import-execute}" method="post"
      th:object="${confirmForm}"
      onsubmit="return confirm('Replace all operational data? This cannot be undone.');">
  <input type="hidden" th:field="*{stagingId}">
  <div class="form-group">
    <div class="form-check">

      </label>
    </div>

           th:errors="*{acknowledged}" class="field-error">
      You must acknowledge the deletion warning to continue.
    </small>
  </div>
  <div class="actions">
    <a th:href="@{/admin/backup/import-cancel(stagingId=${confirmForm.stagingId})}"
       class="btn btn-secondary">Cancel</a>
    <button type="submit" class="btn btn-danger">Execute Import</button>
  </div>
</form>
```

**Layout decisions:**
- **Warning callout first, recap second, checkbox third, buttons last.** Top-to-bottom F-scan order matches operational severity: red flag → context → user action → submit.
- **`.form-check` flex pattern** (`admin.css:335`) puts the checkbox and label on one line with 8px gap. Standard HTML — works with screen readers via the `<label for="acknowledged">` association.
- **`.btn-danger` ONLY on the Execute Import button** (CLAUDE.md no-inline-styles + 10 %-accent rule). Cancel is `.btn-secondary`. No accent colour on this page.
- **JS `confirm()` is UX-extra** (per CONTEXT D-10). If JS is disabled, the form still POSTs; the server-side `@AssertTrue` on `acknowledged` is the authoritative gate. If the checkbox is unchecked, the page re-renders with the `.field-error` message inline next to the checkbox.

### Screen-reader behaviour on Flash redirects

The layout (`admin/layout.html:82-83`) renders Flash messages as `<div class="alert alert-success">` / `<div class="alert alert-error">` at the **top of `<main>`**, before the page heading. Recommend planner adds `role="status"` to `alert-success` and `role="alert"` to `alert-error` on the layout — that's a 1-line layout change but it's a Phase-74-touched file so the planner can ship it as part of this phase. Alternatively, defer to a follow-up phase if `layout.html` is treated as do-not-touch.

### Mobile/Desktop responsiveness

**`.card-grid` (`admin.css:1692-1696`):**
```css
.card-grid {
    display: grid;
    grid-template-columns: repeat(auto-fill, minmax(220px, 1fr));
    gap: 16px;
}
```

This auto-fills:
- **Mobile (< 600px container)**: 1 column (220px minmax always fits)
- **Tablet (600-900px)**: 2-3 columns
- **Desktop (1200px container max-width minus 64px container padding = ~1136px)**: 5 columns of 220px + gaps

At 24 cards on desktop, this renders as ~5 rows. At mobile, this is 24 rows — long scroll, but each card is full-width and tappable. Phase 74 does NOT add a responsive override; the existing breakpoint behaviour is correct.

**Sidebar at mobile:** the existing `.sidebar-toggle` button + `.sidebar-overlay` handle the off-canvas pattern; Phase 74 does nothing new here.

**Touch targets:** `.btn` minimum height is 36px (`admin.css:216`), and `.btn-lg` is 48px (12px padding + 24px font line-height). The file input and checkbox inherit native browser touch sizes (≥ 44px on iOS/Android). All meet the 44px-touch-target recommendation.
