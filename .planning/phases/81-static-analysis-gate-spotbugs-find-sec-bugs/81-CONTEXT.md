# Phase 81: Static Analysis Gate (SpotBugs + find-sec-bugs) - Context

**Gathered:** 2026-05-16
**Status:** Ready for planning

<domain>
## Phase Boundary

Wire `spotbugs-maven-plugin` 4.9.8.3 with the `findsecbugs-plugin` 1.14.0 plugin-dependency into the main `<build><plugins>` block of `pom.xml`, bound to the `verify` phase **after** the existing JaCoCo `check` execution. Land `lombok.config` at project root and `config/spotbugs-exclude.xml` so Lombok-generated bytecode and intentional hardening patterns (SSRF allowlist, ZIP-Slip defense, BCrypt encoder bean) are pre-suppressed before the gate goes blocking. Execute the staged hand-off mandated by STAT-05: first commit configures `<goal>spotbugs</goal>` (report-only), interleaved triage commits fix HIGH findings and document Medium suppressions, final commit flips to `<goal>check</goal>` so `./mvnw verify` fails the build on new Medium+HIGH violations. JaCoCo argLine late-evaluation stays untouched (Pitfall #7).

**In scope:**
- `lombok.config` at project root (project does NOT yet have one) with two lines: `lombok.addLombokGeneratedAnnotation = true` + `lombok.extern.findbugs.addSuppressFBWarnings = true` (STAT-01)
- `spotbugs-maven-plugin` 4.9.8.3 in main `<build><plugins>` block, declared AFTER `jacoco-maven-plugin` (STAT-02)
- `findsecbugs-plugin` 1.14.0 as `<plugins>` dependency of the SpotBugs plugin (STAT-03) — all 144 patterns active
- `config/spotbugs-exclude.xml` with hybrid exclusion scope (3 class-level excludes + EI_EXPOSE_REP* pattern on `org.ctc.domain.model.*`) and rationale comments on every entry (STAT-04)
- Two atomic STAT-05 commits with an interleaved triage commit-group between them
- HIGH SpotBugs findings (Medium+HIGH after Lombok mitigation) fixed in-phase; Medium findings triaged (real bug → fix, intentional pattern → suppress + rationale + code-comment cross-ref, stylistic → suppress with short reason)
- `STAT-06` validation: throwaway-branch deliberate-violation test that proves the gate blocks
- CLAUDE.md "Conventions" section update documenting the gate + suppression-file workflow + `lombok.config` invariant (STAT-07)

**Out of scope (deliberate):**
- Checkstyle / PMD integration — explicitly rejected per Research Conflict 2 (Lombok-heavy single-team project; CLAUDE.md already enforces naming + Java 25 `-Xlint` already covers PMD-overlap rules)
- Profile-based plugin invocation (the `rewrite` profile pattern from Phase 80 is the OPPOSITE shape — SpotBugs is verify-bound, always-on)
- CI-level `dockerfile-noble-pin-guard`-style separate workflow — SpotBugs runs inside existing `verify`, no new CI job needed
- Custom SpotBugs detector classes — only standard SpotBugs + find-sec-bugs patterns
- Modifying JaCoCo plugin configuration (Pitfall #7 — no plain `<argLine>` ever, `@{argLine}` stays untouched)
- Changing existing JaCoCo `<excludes>` list (14 classes) — see Decision D-08 for why SpotBugs exclusion scope deliberately diverges from coverage exclusion scope

<decisions>
## Implementation Decisions

### Plugin Wiring (pom.xml)

- **D-01:** `spotbugs-maven-plugin` declared in the **main `<build><plugins>`** block (not in a `<profile>` — SpotBugs must run on every `./mvnw verify`, unlike OpenRewrite which is developer-invoked-only). Insertion point: directly after the existing `jacoco-maven-plugin` block (pom.xml:303-355) and before the `exec-maven-plugin` template-fragment-call-guard block (pom.xml:356-384) so the execution order is `failsafe:integration-test` → `jacoco:report` → `jacoco:check` → `failsafe:verify` → `spotbugs:check`.
- **D-02:** Plugin version: defer final pin to the planner. Research-verified is 4.9.8.3 (Maven Central 2026-05-16). If a newer stable patch exists at planning time and changelog shows no behavior-breaking changes, prefer the newer patch; otherwise pin 4.9.8.3.
- **D-03:** `findsecbugs-plugin` 1.14.0 declared as a `<plugins>` (not `<dependencies>`) child of the SpotBugs plugin block — this is SpotBugs' specific extension mechanism for detector packs. Same pin-defer rule as D-02.
- **D-04:** **No `<argLine>` on the SpotBugs plugin block, ever** (Pitfall #7). SpotBugs is a bytecode scanner, not a JVM agent — JaCoCo's `@{argLine}` late-evaluation must remain the single source of `argLine` truth. Plan must include a post-wiring smoke-check that `target/site/jacoco/index.html` shows non-zero coverage numbers (>= 82%, ideally >= 87.80% v1.10 baseline).

### Effort & Threshold

- **D-05:** **`<effort>Max</effort>`** — full bytecode analysis depth (STAT-02). Adds ~30-90s to `mvn verify` (acceptable; total verify currently sits comfortably under the CI budget).
- **D-06:** **`<threshold>Default</threshold>`** for BOTH the report goal (`spotbugs:spotbugs`) and the blocking goal (`spotbugs:check`) — Medium+HIGH findings get reported AND block the build. Strict reading of STAT-02. SC#4 ("HIGH-priority") is interpreted as the strictest realistic minimum threshold, NOT a ceiling — blocking Medium too is more pedantic and consistent with the verify-time always-on gate philosophy. Trade-off accepted: Initial cleanup workload is higher because Medium find-sec-bugs findings (e.g. `PREDICTABLE_RANDOM`, `WEAK_FILENAMEUTILS` if it fires) must all be either fixed or suppressed-with-rationale before the gate flips to `check`.
- **D-07:** `<failOnError>true</failOnError>` on the `check` goal execution (default behavior; called out explicitly so the planner doesn't have to research it).

### Exclusion Scope (`config/spotbugs-exclude.xml`)

- **D-08:** **Hybrid exclusion strategy** — SpotBugs scope deliberately diverges from JaCoCo exclude list because the two questions are orthogonal (coverage = "do we test it" vs static-analysis = "could it harbor bugs"). Three layers:
  1. **Class-level excludes (3 only):**
     - `org.ctc.CtcManagerApplication` — Spring Boot main, no business logic
     - `org.ctc.admin.TestDataService` — `@Profile("dev")` test-data provider, e2e-tested only
     - `org.ctc.admin.DemoDataSeeder` — `@Profile("dev,demo")` demo-data provider, manual-test-only
  2. **Pattern-on-package exclude:** `EI_EXPOSE_REP` + `EI_EXPOSE_REP2` on `org.ctc.domain.model.*` (covers all 24 JPA entities — Pitfall #4). This is belt-and-braces with `lombok.extern.findbugs.addSuppressFBWarnings=true` because the Lombok flag emits `@SuppressFBWarnings` at the method level, but the package-level filter catches edge cases where Lombok cannot annotate (e.g., synthetic accessor methods).
  3. **NO class-level excludes** for the 11 Playwright graphic services (`TeamCardService`, `LineupGraphicService`, `ResultsGraphicService`, `SettingsGraphicService`, `OverlayGraphicService`, `MatchResultsGraphicService`, `PowerRankingsGraphicService`, `AbstractGraphicService`, `PlayoffRoundOverviewGraphicService`, `PlayoffRoundScheduleGraphicService`, `PlayoffRoundResultsGraphicService`) — they ARE production code; bugs there CAN appear in generated PNGs. If specific Medium findings fire on them (e.g., `DM_DEFAULT_ENCODING` on Playwright HTML emission, `OS_OPEN_STREAM` on a Page.close path), suppress per-method or per-pattern with rationale in `spotbugs-exclude.xml` — never blanket-class-level.

- **D-09:** **Every entry in `spotbugs-exclude.xml` MUST carry an XML comment with rationale and (where applicable) a code-comment cross-reference** (STAT-04 + "no blanket SuppressWarnings"). Example:
  ```xml
  <!-- FileStorageService.storeFromUrl(): SSRF hostname blocklist (Set.of(...)) is
       intentional defense; find-sec-bugs cannot recognize it as a sanitizer.
       See FileStorageService:NN comment block. -->
  <Match>
    <Class name="org.ctc.domain.service.FileStorageService"/>
    <Method name="storeFromUrl"/>
    <Bug pattern="SSRF"/>
  </Match>
  ```
  The plan should add a build-guard check (similar to the existing `template-fragment-call-guard` `exec-maven-plugin` block) that asserts every `<Match>` element in `config/spotbugs-exclude.xml` has a non-empty preceding XML comment. **Discretion:** if the build-guard adds visible CI cost or is fragile against XML formatting, drop it and rely on code-review discipline instead.

### Fix-vs-Suppress Posture

- **D-10:** **Hybrid posture for findings beyond Lombok false positives.** Decision tree per finding:
  1. **HIGH-priority** → MUST fix in this phase (atomic commit per pattern-family or per file). No suppressions of HIGH findings except for the documented intentional-hardening trio (SSRF allowlist, ZIP-Slip defense, BCrypt encoder bean) — and those are suppressed with code-cross-referenced rationale.
  2. **Medium-priority, real bug** → fix in this phase (atomic commit).
  3. **Medium-priority, intentional pattern** (project-specific hardening that find-sec-bugs cannot recognize) → suppress with rationale comment + code-comment cross-reference (D-09 format).
  4. **Medium-priority, stylistic / low-impact** (e.g., `DM_DEFAULT_ENCODING` on log files where charset is implicit-UTF-8 on all production JVMs) → suppress with short rationale.
  5. **Anything that cannot be triaged** → escalate to discussion, do NOT default-suppress.

- **D-11:** **Anticipated intentional-pattern suppressions** (planner should pre-stage these in `spotbugs-exclude.xml` so the report-only run already covers them, but verify with the actual baseline report before committing):
  - `SSRF` on `FileStorageService.storeFromUrl` (Pitfall #12 — `Set.of(...)` hostname blocklist not recognized as CodeQL/FSB sanitizer)
  - `PATH_TRAVERSAL_IN` on `BackupImportService.restoreOneTable` (Pitfall #14 — `startsWith(uploadDir.toRealPath())` defense not recognized as FSB sanitizer)
  - `PATH_TRAVERSAL_IN` on `FileStorageService.store*` (same shape as above)
  - `HARD_CODE_PASSWORD` on `SecurityConfig.passwordEncoder` bean method (Pitfall #13 — `BCryptPasswordEncoder(strength)` keyword-match false positive)
  - `EI_EXPOSE_REP*` already covered by D-08 layer 2

### Two-Commit STAT-05 Choreography

- **D-12:** Commit sequence on the phase feature branch (off `origin/master` per CLAUDE.md):
  1. **Plumbing commit:** `lombok.config` (project root) + `config/spotbugs-exclude.xml` (with D-11 pre-staged intentional suppressions + D-08 Lombok package-level filter) + pom.xml plugin block configured with `<goal>spotbugs</goal>` (report-only, NOT `check`). After this commit, `./mvnw verify` runs SpotBugs in report-only mode and produces `target/spotbugsXml.xml`. Message: `feat(81): wire SpotBugs report-only baseline (STAT-01..STAT-04, STAT-05/1)`.
  2. **Baseline inspection (no commit):** Run `./mvnw verify` once, capture `target/spotbugsXml.xml` + `target/site/spotbugs.html`. Count findings. Decide per D-10 decision tree which findings need code fixes vs additional suppressions. Document this triage table in `81-VERIFICATION.md` (created by the planner, populated by the executor).
  3. **Triage commits (N commits, one per pattern-family or per file):** Real-bug fixes in source code, OR new `<Match>` entries in `spotbugs-exclude.xml` with rationale (D-09 format). Each commit message references the affected pattern/class. After all triage commits, `mvn verify` (still report-only goal) must produce ZERO Medium+HIGH findings in the report.
  4. **Gate-flip commit:** Single-line pom.xml change from `<goal>spotbugs</goal>` to `<goal>check</goal>`. After this commit `./mvnw verify` blocks on Medium+HIGH violations. Message: `feat(81): activate SpotBugs blocking gate (STAT-05/2, STAT-06)`.

- **D-13:** **STAT-06 deliberate-violation test is performed on a throwaway branch and reverted before merging the phase PR.** The plan must include a `81-VERIFICATION.md` step:
  1. `git switch -c throwaway/stat-06-validation`
  2. Introduce a deliberate HIGH-priority violation (e.g., `Object o = null; o.toString()` in a fresh test-only file under `src/main/java/org/ctc/_validation_marker/`).
  3. Run `./mvnw verify` → must fail with SpotBugs `NP_ALWAYS_NULL` or equivalent. Capture exit code + first 20 log lines of the failure into `81-VERIFICATION.md`.
  4. `git switch -` and delete the throwaway branch. No artifact from the throwaway branch lands in `origin`.

### Documentation (STAT-07)

- **D-14:** CLAUDE.md gets a new paragraph in the **`## Conventions`** section (after the existing CSS-Guidelines line). Three points:
  1. SpotBugs + find-sec-bugs gate active on every `./mvnw verify` (Medium+HIGH block the build).
  2. Suppressions live in `config/spotbugs-exclude.xml`; every `<Match>` requires an XML rationale comment + code-comment cross-reference where applicable; **no `@SuppressWarnings("all")` ever**.
  3. `lombok.config` at project root is an invariant — do NOT remove or modify the two SpotBugs-related lines without a new phase that re-baselines suppressions (would re-introduce ~40-80 `EI_EXPOSE_REP*` Lombok false positives).
- **D-15:** Discretion: README does NOT need an OpenRewrite-style "Development" subsection for SpotBugs — SpotBugs is invisible-by-default (just part of `mvn verify`). If the planner finds a natural spot in README's existing developer-workflow section, a one-liner is fine but not required. CLAUDE.md is the authoritative location per STAT-07.

### Claude's Discretion

- Final plugin version pins for `spotbugs-maven-plugin` (D-02) and `findsecbugs-plugin` (D-03) — pick latest stable on Maven Central at planning time.
- The exact file layout of `config/spotbugs-exclude.xml` (alphabetical vs by-package vs by-pattern grouping) — match the readability that suppression-count requires.
- Whether the deliberate-violation throwaway branch (D-13) lives in `src/main/java/` or `src/test/java/` — depends on which one triggers `spotbugs:check` in the current pom.xml layout (main is the default; test only if `<includeTests>true</includeTests>` is set, which D-05 does NOT set).
- Whether the suppression-rationale build-guard (D-09 tail) is worth its CI weight — the planner inspects the existing `template-fragment-call-guard` pattern and decides.
- Suppression-rationale wording — match the existing CLAUDE.md tone (terse, English, with file:line references).
- Whether `<plugins>` needs to be alphabetized in pom.xml after insertion (current pom.xml: surefire/failsafe/jacoco/exec/versions are roughly grouped by function not alphabetical; mimic that).

</decisions>
