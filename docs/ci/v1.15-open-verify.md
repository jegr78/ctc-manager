## CI-01 / CI-02 — docs-only required-checks success + CodeQL/MariaDB absent

**Config verified correct:**

- `ci.yml` `changes` job (`dorny/paths-filter@v4.0.1`) splits `code` (src, `pom.xml`,
  `.github/workflows/**`, Dockerfile, mvnw, …) from `docs` (`.planning/**`, `*.md`, `docs/**`,
  `.gitmessage`). The three required jobs gate every real step with
  `if: needs.changes.outputs.code == 'true'`, while checkout + `if: always()` uploads run
  unconditionally — so each reports **success** (not `skipped`) on a docs-only run.
- `codeql.yml` and `mariadb-migration-smoke.yml` carry matching path filters
  (`paths-ignore: ['.planning/**','*.md','docs/**','.gitmessage']` and a positive `paths:`
  restricted to migrations/pom/application-yml respectively) that exclude docs-only diffs.

**Empirical caveat observed (cumulative PR-diff semantics):** a docs-only *commit* pushed onto
the milestone PR `#132` did **not** isolate docs-only behaviour — `CodeQL SAST`, `MariaDB
Migration Smoke`, and the full `build-and-test` all ran. This is correct GitHub behaviour, not a
defect: for `pull_request` events the path filters (both GitHub's native `paths`/`paths-ignore`
and `dorny/paths-filter`) evaluate the **cumulative PR diff** (`base…head`), which on `#132`
contains source + migrations. The docs-only short-circuit therefore manifests only for a PR whose
**entire** diff is docs-only (e.g. a standalone README/wiki PR), or a docs-only push to `master`.

**Status: CI-01/02 accepted as config-sound.** The path filters are verified correct by
inspection; the cumulative-diff caveat is inherent GitHub semantics. A dedicated docs-only PR was
deliberately not opened (would add throwaway PR churn for no additional config confidence). The
behaviour can be confirmed opportunistically on the next genuine docs-only PR.
